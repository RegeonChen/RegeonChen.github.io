## 〇、一个 C 程序的一生 —— 八章内容的完整串联

让我们从一个最简单的 C 程序出发，追踪它从源代码到运行、再到与其它程序共存的完整生命周期。这恰好能把第 1–9 章（除第 4 章）的知识全部串联起来。

假设我们有这样一个程序 `hello.c`：

```c
#include <stdio.h>
int global = 42;           // 已初始化的全局变量
int uninit;                // 未初始化的全局变量

int main() {
    int local = 10;        // 局部变量
    printf("hello, %d\n", global + local);
    return 0;
}
```

---

### 故事一：从源代码到可执行文件 —— 编译四阶段

#### 第 1 步：预处理（Preprocessing）— `hello.c` → `hello.i`

预处理器 `cpp` 处理以 `#` 开头的指令：
- `#include <stdio.h>` → 把头文件内容**文本插入**到当前文件
- `#define` 宏展开、条件编译 `#ifdef` 等

输出：`hello.i`，一个**展开后的纯 C 文本文件**（仍然是文本）。

> **关联章节**：第 1 章（导论 §1.2 编译系统）

---

#### 第 2 步：编译（Compilation）— `hello.i` → `hello.s`

编译器 `cc1` 把 C 代码翻译成**汇编代码**。这个过程涉及：
- 词法分析、语法分析、语义分析
- **数据类型**的分配：`int global = 42` 中的 `42` 被翻译成 `.long 42`（第 2 章数据表示决定了 `int` 占 4 字节、42 的补码是 `0x0000002A`）
- **局部变量** `local = 10` 被分配到寄存器或栈上（第 3 章：`movl $10, -4(%rbp)`）
- **函数调用** `printf(...)` 被翻译成参数准备 + `call printf`（第 3 章：ABI 约定，`%rdi` 放第一个参数——格式化字符串地址）

输出：`hello.s`，**汇编代码文本文件**。

> **关联章节**：第 2 章（数据如何编码为立即数）、第 3 章（C 语句到汇编指令的映射）

---

#### 第 3 步：汇编（Assembly）— `hello.s` → `hello.o`

汇编器 `as` 把汇编代码翻译成**机器码**，但此时地址还是"占位符"：

- `movl $10, -4(%rbp)` → 翻译为对应的机器码字节（如 `c7 45 fc 0a 00 00 00`）
- `call printf` → 翻译为 `e8 00 00 00 00`（`00 00 00 00` 是占位符，因为还不知道 `printf` 的地址）
- `global`、`uninit`、`printf` 等符号写入**符号表** `.symtab`
- 需要重定位的引用写入**重定位表** `.rel.text` 和 `.rel.data`

输出：`hello.o`，**ELF 可重定位目标文件**（二进制），结构包含：

| 段 | 内容 | 关联章节 |
|----|------|----------|
| `.text` | `main` 函数的机器码 | 第 3 章（汇编→机器码） |
| `.data` | `global` 的初始值 `0x0000002A`（42）| 第 2 章（小端序存储为 `2A 00 00 00`）|
| `.bss` | `uninit` 占位（只记录大小 4 字节，不占磁盘空间）| 第 7 章 |
| `.rodata` | 字符串常量 `"hello, %d\n"` | 第 7 章 |
| `.symtab` | 符号表：定义了 `main`、`global`、`uninit`，引用了 `printf` | 第 7 章 |
| `.rel.text` | 需要重定位的地方（`call printf` 的地址、`global` 的地址）| 第 7 章 |

> **关联章节**：第 2 章（数据的字节级编码）、第 3 章（汇编指令→机器码编码）、第 7 章（ELF 格式）

---

#### 第 4 步：链接（Linking）— `hello.o` + `libc.so` → `hello`（可执行文件）

链接器 `ld` 做两件事：

**① 符号解析**：把每个符号引用关联到唯一定义。
- `printf` 在 `hello.o` 中是"未定义"的 → 链接器去 `libc.so` 中找到 `printf` 的定义
- `global` 在 `hello.o` 中是"定义"的 → 作为强符号
- `uninit` 在 `hello.o` 中是"定义"的弱符号 → 合并到 `.bss`

**② 重定位**：
- 合并所有输入模块的 `.text` 段
- 把 `call printf` 后面的 `00 00 00 00` 替换为 `printf` 的真实偏移地址
- 把 `global` 的引用替换为运行时虚拟地址

此时，如果是**动态链接**（默认），`printf` 不会真正被填进可执行文件——只会在 GOT/PLT 中留下"入口桩"（stub），等运行时第一次调用时再由动态链接器 `ld-linux.so` 解析（延迟绑定，lazy binding → 第 7 章）。

输出：`hello`，**ELF 可执行文件**，它的虚拟地址空间布局已经由链接器决定了（`0x400000` 附近开始）。

> **关联章节**：第 7 章（整个链接过程）、第 3 章（重定位修改的是哪些指令）

### 故事二：数据的一生 —— 不同类型的数据在存储金字塔中的流动

当我们在终端敲下 `./hello`，数据就开始在整个存储金字塔中流动了。

#### 加载时：程序映像从磁盘"进入"主存

操作系统**不会**把整个 `hello` 可执行文件拷进物理内存，而是通过**虚拟内存**（第 9 章）做"懒加载"：

| 段 | 数据类型 | 后备存储 | 初始加载时发生了什么 | 关联章节 |
|----|---------|----------|-------------------|----------|
| `.text` | 代码（机器指令）| 可执行文件 `hello`（磁盘上的持久化文件）| 建立页表映射，标记为"不在物理内存"，第一次执行时触发**缺页故障**（第 8 章§8.1 故障），从磁盘调入物理内存 | 第 7、8、9 章 |
| `.rodata` | `"hello, %d\n"` 字符串常量 | 同上 | 同上，缺页时从磁盘读入 | 第 7、9 章 |
| `.data` | `global = 42`（`0x0000002A`）| 同上 | 同上，缺页时从磁盘读入；后续可能被修改 → 变成**脏页** | 第 2、7、9 章 |
| `.bss` | `uninit`（初值 0）| Swap Space（匿名页）| 映射到全局零页（全零物理页），第一次写入时 COW 分配私有页 | 第 7、9 章 |
| 栈 | `local = 10`、返回地址等 | Swap Space（匿名页）| OS 在 `execve` 时创建初始栈帧，缺页时从 swap 换入（第一次访问时实际分配零页）| 第 3、8、9 章 |
| 堆 | `malloc()` 分配的内存 | Swap Space（匿名页）| 初始未分配，`brk`/`mmap` 扩展；缺页时从 swap 换入 | 第 9 章 |

**关键机制**：无论是文件后备页（`.text`、`.data`）还是匿名页（栈、堆、`.bss`），都是**第一次被 CPU 访问时才真正调入物理内存**——这个"第一次访问 → 缺页 → 调入"的流程，正是第 8 章的**故障异常**加上第 9 章的**缺页处理**协同完成的。

#### 运行时：数据在存储金字塔的各级之间移动

```
CPU 执行 movl global, %eax  欲读取 global 的值 42
   │
   ▼
第1步：CPU 发出虚拟地址（.data 段中 global 的地址）
   │
   ▼
第2步：查 TLB（Translation Lookaside Buffer，第9章）
   │  TLB 是页表的硬件缓存（SRAM，L1 级别速度）
   │
   ├── TLB 命中 → 直接拿到物理页号
   │
   └── TLB 不命中 → 查页表（页表本身在物理内存 DRAM 中）
         │
         ├── 页表 PTE 有效位=1 → 拿到物理地址
         │
         └── 页表 PTE 有效位=0 → 缺页故障！
              操作系统：
              ├── 文件后备页 → 从磁盘（可执行文件）读入物理内存
              ├── 匿名页     → 分配零页 / 从 swap 换入
              └── 更新页表 PTE，返回重试
   │
   ▼
第3步：拿到物理地址后，查 L1 数据缓存（第6章）
   │
   ├── L1 命中（~4 周期）→ 返回 42 给 %eax
   │
   ├── L1 不命中 → 查 L2 缓存（~10 周期）
   │     ├── L2 命中 → 装入 L1 + 返回
   │     └── L2 不命中 → 查 L3 缓存（~40 周期）
   │           ├── L3 命中 → 装入 L2 + L1 + 返回
   │           └── L3 不命中 → 访问主存 DRAM（~200 周期）
   │                 └── DRAM 返回 42 → 沿路填回 L3/L2/L1 → 给 %eax
   │
   ▼
第4步：%eax 拿到 42，addl local, %eax → 结果写入寄存器
```

**四种不同类型数据的流动路径总结**：

| 数据类型 | 典型例子 | 流动路径 | 关联章节 |
|----------|---------|----------|----------|
| **代码**（指令）| `movl`、`call` 指令的机器码 | 磁盘（ELF）→ 主存（缺页调入）→ L1-I 指令缓存 → CPU 取指 | 第 3、6、7、9 章 |
| **只读数据** | `"hello, %d\n"` 字符串、`const` 变量 | 磁盘（ELF）→ 主存 → L1/L2/L3 数据缓存 → 寄存器 | 第 2、6、7、9 章 |
| **全局可读写数据** | `global = 42`（`.data`）、`uninit`（`.bss`）| `.data`：磁盘→主存→缓存→寄存器；`.bss`：零页→COW 后私有页→缓存→寄存器；修改后变脏页，可能被换出到 swap | 第 2、6、7、9 章 |
| **栈上局部变量** | `local = 10`、返回地址 | 匿名页（swap）→ 主存 → L1/L2/L3 缓存 → 寄存器；高频读写，通常在 L1 中 | 第 3、6、9 章 |
| **动态分配的堆内存** | `malloc(sizeof(int))` | 匿名页（swap）→ 主存 → 缓存 → 寄存器 | 第 9 章 |

**一条数据指令的完整旅行**（以 `global = 42` 为例）：

```
磁盘上的 ELF 文件
   │  .data 段中 4 个字节：2A 00 00 00（小端序，第2章）
   │
   ▼ (execve，第8章：替换进程映像)
虚拟地址空间（第9章：.data 段映射到某虚拟地址）
   │
   ▼ (第一次访问 global 时，CPU 发出虚拟地址)
TLB 查询（第9章：快表）
   │
   ▼ (TLB miss → 查页表 → PTE 无效 → 缺页故障，第8章§8.1)
OS 缺页处理（第9章）→ 从磁盘 ELF 文件读入 4KB 页面到物理内存（DRAM，第6章）
   │
   ▼ (页面在 DRAM 中，后续访问走缓存)
L3 → L2 → L1 缓存（第6章）→ 寄存器 %eax（第3章）
   │
   ▼ (addl local, %eax → global + local = 52)
ALU 计算 42 + 10 = 52 → %eax = 52
   │
   ▼ (call printf → 第3章 ABI：%rdi=字符串指针，%rsi=52)
printf 内部将 52 格式化为字符 '5' '2'，写入 stdout 缓冲区
   │
   ▼ (write 系统调用 → syscall 指令，第8章§8.1 陷阱异常)
内核将缓冲区数据通过网络或终端驱动输出到屏幕
```

### 故事三：多个程序同时运行 —— OS 的角色

如果有两个用户同时在运行程序（或者一个用户开了两个终端），操作系统需要同时管理 `hello`、`bomb`、shell 等多个进程。

#### 1. 每个进程拥有独立的虚拟地址空间

即使 `hello` 和 `bomb` 都用到了地址 `0x4004d0`（它们的 `.text` 入口），它们指向的物理页完全不同——因为每个进程有自己独立的页表（第 9 章）。上下文切换时，内核切换 CR3 寄存器，指向新进程的页表基址（第 8 章§8.2）。

> **这是第 9 章最重要的设计目标：让每个进程以为自己独占内存。**

#### 2. 共享只读页面：节省物理内存

如果 `hello` 和另一个进程都链接了 `libc.so`，`libc.so` 的 `.text` 段只会加载**一份**物理副本（第 7 章 PIC + 第 9 章共享内存映射）。两个进程的页表都把对应的虚拟地址映射到**同一块物理页**。

> **这里第 7 章（PIC）和第 9 章（mmap 共享映射）协同工作。**

#### 3. 上下文切换：让 CPU "分身"

物理 CPU 只有一个（或几个），但 OS 让每个进程感觉独占 CPU：

```
时间 →
进程A(hello): [运行]──[缺页等待]──────[运行]──[I/O等待]────────[运行]
进程B(bomb):  ──[就绪]────────[运行]────────────[运行]──[就寝]────
进程C(shell): ────[就寝]──────────────────────[运行]────────[就寝]

内核:
  时钟中断（第8章§8.1 中断）→ scheduler 决定切换 → 保存A的寄存器
  → 恢复B的寄存器 → B运行 → 时钟中断 → 保存B → 恢复A → A继续...
```

每个切换点上，内核做了（第 8 章§8.2）：
1. **保存**当前进程的寄存器状态（`%rax`、`%rsp`、`%rip` 等，这些寄存器的含义见第 3 章）
2. **切换**页表（CR3 指向新进程的页表基址，第 9 章）
3. **恢复**新进程的寄存器状态
4. `iret` 返回用户态

#### 4. `fork()` + `execve()`：新进程的诞生

当用户在 shell 中敲下 `./hello`，shell 实际上做了：

```c
pid_t pid = fork();                     // 第8章§8.4
if (pid == 0) {                         // 子进程
    execve("./hello", argv, envp);      // 第8章§8.5
}
waitpid(pid, &status, 0);               // 第8章§8.5
```

- **`fork()`**：创建父进程的副本。关键优化——**写时复制 COW**（第 9 章）：`fork()` 后父子共享所有物理页，页表标记为只读。任一方写入时 → 保护故障（第 8 章§8.1 故障）→ 内核复制一页私有副本（第 9 章）。这就是为什么我们常说"fork 很快"——它并不真的复制所有内存，只是复制了页表，真正的拷贝按需发生。
- **`execve("./hello", ...)`**：不创建新进程，而是替换当前进程的映像。内核：
  1. 释放旧页表（第 9 章）
  2. 解析 ELF 文件（第 7 章），建立新的虚拟地址空间映射（第 9 章）
  3. 设置栈、初始化寄存器（第 3 章：`%rsp` 指向新栈顶）
  4. 跳转到 `_start` → `__libc_start_main` → `main`（第 7 章：`_start` 由 crt1.o 提供，链接时进入）

#### 5. 信号：进程间的异步通知

当用户按下 `Ctrl-C`：
1. 键盘中断 → 内核检测到 `SIGINT`（第 8 章§8.6）
2. 内核向**前台进程组**发送 `SIGINT`（不是发给 shell，是发给正在运行的 `hello`）
3. `hello` 如果注册了 handler → 调用 handler；如果没注册 → 默认行为：终止进程
4. 子进程终止 → 内核向父进程（shell）发 `SIGCHLD` → shell 调用 `waitpid` 回收子进程（第 8 章§8.5）

#### 6. 总结：OS 在多个程序同时运行时的核心职责

| OS 职责 | 使用的机制 | 关联章节 |
|---------|-----------|----------|
| **隔离**：每个进程以为独占内存 | 每进程独立页表 + 虚拟地址空间 | 第 9 章 |
| **调度**：共享 CPU | 时钟中断 → 上下文切换 → 保存/恢复寄存器 | 第 8 章§8.1-§8.2、第 3 章（寄存器）|
| **通信**：进程间通知 | 信号（`SIGCHLD`、`SIGINT` 等）| 第 8 章§8.6 |
| **效率**：节省内存 | COW（`fork`）+ 共享库映射（PIC + mmap）| 第 8 章§8.4 + 第 9 章 + 第 7 章 |
| **安全**：进程不可越界 | SIGSEGV + 页表权限位（R/W/U）+ 内核空间保护 | 第 8 章§8.6 + 第 9 章 |

## 逐章详解 + 前驱后继关系

### 第 1 章 · 导论 —— 给一张地图

**核心问题**：`hello.c` 从敲下到屏幕输出到底发生了什么？

| 知识点 | 细节 |
|--------|------|
| 信息的"位+上下文"表示 | 同一串 01 在不同上下文中（整数/浮点/指令）含义完全不同 |
| C 程序编译四阶段 | 预处理 → 编译 → 汇编 → 链接（`gcc -E/-S/-c/-o`） |
| 硬件组成 | CPU（PC + ALU + 寄存器）、主存、总线、I/O 设备 |
| 存储金字塔概念 | 寄存器 → L1 → L2 → L3 → 主存 → 本地磁盘 → 远程存储 |
| 操作系统三大抽象 | **进程**（CPU 抽象）、**虚拟内存**（主存抽象）、**文件**（I/O 抽象） |
| Amdahl 定律 | 加速比受限于可并行部分的比例 |

### 第 2 章 · 数据表示 —— 一切计算的根基

**核心问题**：同一个 32 位 `0xFFFFFFFF`，为什么有时是 `-1`，有时是 `4294967295`，有时是 `NaN`？

**关键知识点**：
- 大端/小端（数据在内存中的字节序）
- 无符号 vs 补码（B2U / B2T）
- 隐式类型转换的坑（有符号+无符号 → 无符号；`sizeof` 返回 `size_t` 的 bug）
- 整数溢出（截断，不是 UB）
- 位扩展（零扩展 / 符号扩展）与位截断
- 移位操作：除法用移位优化（~1 周期 vs ~30 周期）
- 浮点数 IEEE 754（符号+阶码+尾数；规格化/非规格化/特殊值）

### 第 3 章 · 汇编 —— 连接 C 与硬件的桥梁

**核心问题**：`int sum(int a, int b) { return a + b; }` 在 CPU 眼里到底是什么？

**关键知识点**：
- ISA vs 微架构 vs OS：软件兼容边界、硬件实现、操作系统管理
- x86-64 寄存器：`%rax`（返回值）、`%rdi/%rsi`（第 1/2 参数）、`%rdx/%rcx`（第 3/4 参数）、`%rsp`（栈顶指针）、`%rip`（PC）
- 操作数寻址模式：立即数 `$`、寄存器 `%r`、内存 `Imm(%rb,%ri,s)` —— 这是最核心的
- 数据传送指令：`movb/w/l/q`、`movz`（零扩展）、`movs`（符号扩展）；`pushq`/`popq`
- 算术与逻辑指令：`add/sub/inc/dec`、`and/or/xor`、`shl/shr/sar`
- 条件码（Condition Codes）：`CF/ZF/SF/OF`，由 `cmp`/`test` 设置
- 条件跳转与控制流：`jmp`/`je`/`jne`/`jg`/`jl`...；`cmov` 条件传送
- 过程调用：`call`（压返回地址+跳转）/`ret`（弹出返回地址+跳转）；栈帧（`%rsp`/`%rbp`）；参数传递约定
- 调用者保存 vs 被调用者保存寄存器
- 数组/结构体/联合体在内存中的布局 + 对齐 + 填充
- 缓冲区溢出原理 → 栈溢出覆盖返回地址
- 栈的成长方向：高地址→低地址（`pushq` 先 `sub $8, %rsp` 再写入）

**前驱后继**：

### 第 5 章 · 优化程序性能 —— 知道原理才能写出快代码

**核心问题**：编译器已经很聪明了，为什么还要程序员来优化？

### 第 6 章 · 存储器层次 —— 为什么计算机能"看起来"内存很快

**核心问题**：SRAM 很快但贵，DRAM 容量大但慢，磁盘更慢——怎么办？

**关键知识点**：
- **局部性原理**：
  - 时间局部性（刚访问的还会再访问）
  - 空间局部性（附近的也会被访问）
- **缓存结构**：(S, E, B, m) — S 组 × E 行/组 × B 字节/块
- **三种映射**：直接映射（E=1）、E 路组相联、全相联（S=1）
- **三种不命中**：冷不命中（第一次访问）、容量不命中（缓存不够大）、冲突不命中（映射到同组）
- **替换策略**：LRU / FIFO
- **写策略**：写直达 / 写回；写分配 / 非写分配
- **缓存友好编程**：行优先遍历（空间局部性）、分块/tiling（时间局部性）

**前驱后继**：
```
第6章 缓存
   │
   ├── 依赖 ← 第2章（块大小 64B 能装 16 个 int 还是 8 个 double？→ 数据宽度）
   │
   └──→ 第9章 虚拟内存：
        ┌─────────────────────────────────────────────┐
        │ 第6章的概念                    第9章的对应     │
        ├─────────────────────────────────────────────┤
        │ 缓存块 (block)        →    页 (page, 4KB)    │
        │ L1/L2/L3 缓存          →    物理内存 (DRAM)   │
        │ 主存 (DRAM)            →    磁盘 (SSD/HDD)    │
        │ 缓存命中/不命中        →    页命中/缺页       │
        │ 写回策略              →    脏页写回           │
        │ LRU 替换              →    页面置换算法       │
        │ 全相联                →    页表是全相联       │

### 第 7 章 · 链接 —— 从碎片到整体

**核心问题**：`main.c` 调了 `printf`，`printf` 在 `libc.so` 里——CPU 怎么找到它的？

**关键知识点**：
- ELF 文件结构：`.text`（代码）、`.rodata`（只读数据）、`.data`（已初始化全局变量）、`.bss`（未初始化全局变量）、`.symtab`（符号表）、`.rel.text` / `.rel.data`（重定位表）
- **符号解析**：
  - 强符号（函数+已初始化全局变量）vs 弱符号（未初始化全局变量）
  - 链接错误类型：重定义、未定义引用
  - `static` 限制作用域
- **重定位**：
  - 合并输入段
  - 修正符号引用（R_X86_64_PC32 相对寻址、R_X86_64_32 绝对寻址）
- **静态库** `.a`：一堆 `.o` 打包，链接器按需提取（只取被引用的模块）
- **动态共享库** `.so`：
  - PIC（Position-Independent Code）：代码可加载到任意地址
  - GOT + PLT：延迟绑定（lazy binding）——首次调用时才解析符号
- 库打桩（library interpositioning）：编译时/链接时/运行时替换库函数

### 第 8 章 · 异常控制流 —— OS 控制中枢

**核心问题**：正常执行的程序，凭什么突然跳去执行内核代码？CPU 从加电到关机的整个指令流中，控制流如何被各种事件打断？

**四个递进层次**：

```
ECF Ⅰ — 硬件异常（§8.1）
      中断（异步，外部 I/O 信号）  → 返回下一条指令
      陷阱（同步，int/syscall）   → 返回下一条指令
      故障（同步，缺页/除零）     → 返回当前指令（修复后重试）或中止
      中止（同步，硬件错误）       → 不返回
      机制：异常表 IDT + IDTR 寄存器
      │
      ▼
ECF Ⅱ — 进程（§8.2-§8.5）
      进程 = 正在运行的程序的实例
      逻辑控制流：即使被切走，似乎独占 CPU
      私有地址空间：每个进程有自己的虚拟地址空间
      fork()   → 创建新进程（复制父进程，写时复制 COW）
      execve() → 替换进程映像（不创建新进程，直接覆盖）
      waitpid() → 回收子进程，防止僵尸进程
      上下文切换 → 内核保存/恢复寄存器状态
      进程状态：运行、就绪、阻塞、终止
      │
      ▼
ECF Ⅲ — 信号（§8.6）
      信号 = 发给进程的"小消息"，一种软件形式的异常
      发送：内核检测到事件 / 其他进程 kill()
      接收：目标进程被强制调用 handler
      待处理 vs 阻塞（pending vs blocked）：信号可以不排队
      SIGCHLD（子进程终止）、SIGINT（Ctrl-C）、SIGSEGV（段错误）等
      sigprocmask 阻塞信号、信号安全的函数列表
      │
      ▼
ECF Ⅳ — 非本地跳转（§8.7）
      setjmp / longjmp → 跨函数直接返回（无需层层 ret）
      sigsetjmp / siglongjmp → 保存/恢复信号掩码
      用途：深层嵌套的错误恢复、signal handler 中跳出
```

```

   │
   ├── 依赖 ← 第3章（syscall 是汇编指令；栈帧布局支持 setjmp/longjmp）
   │
   ├──→ 第9章 虚拟内存（最重要的兄弟关系）：
   │     ┌──────────────────────────────────────────────────────┐
   │     │  fork()  + 写时复制(COW) = 缺页故障驱动的偷懒拷贝       │

### 第 9 章 · 虚拟内存 —— 全书最精妙的抽象

**核心问题**：每个进程都以为自己独占 4GB（或更大）的内存——怎么做到的？

**关键知识点**：
- **虚拟地址 → 物理地址的翻译**：
  - 虚拟地址 = VPN（虚拟页号）+ VPO（页内偏移）
  - 物理地址 = PPN（物理页号）+ PPO（页内偏移）
  - PPO = VPO（偏移不变，只翻译页号）
- **页表**：多级页表（x86-64：四级）
  - 每级 9 位索引 → PTE 8 字节 → 每级 512 条目
  - PTE 包含：有效位、物理页号、权限位（R/W/U）、脏位
- **TLB**：页表的缓存（Translation Lookaside Buffer），命中则单周期完成翻译
- **缺页故障（Page Fault）**：第 8 章的故障异常在此大显身手
  - 页面不在物理内存 → 从后备存储（磁盘/swap）调入
  - 可能先换出脏页（写回磁盘/swap）
- **内存映射（mmap）**：文件直接映射到虚拟地址空间
  - 文件后备页（file-backed）：代码段、只读数据、共享库 → 后备存储是持久化文件
  - 匿名页（anonymous）：堆、栈、`.bss`、COW 产生的私有页 → 后备存储是 swap space
- **COW 写时复制**：`fork()` 的精髓
  - 父子共享同一物理页，标记为只读
  - 任一方写入 → 触发保护故障 → 内核复制一页私有副本
- **进程内存布局**（从低地址到高地址）：
  ```
  0x00000000 → [保留区]
             → .text   （代码段—文件后备，只读）
             → .rodata （只读数据—文件后备）
             → .data   （已初始化全局变量—文件后备）
             → .bss    （未初始化全局变量—匿名，映射到零页）
             → 堆 heap  （malloc/new—匿名，向高地址增长 ↑）
             → [共享库映射区]
             → 栈 stack （局部变量—匿名，向低地址增长 ↓）
  0xFFFFFFFF → [内核空间]（所有进程共享，用户态不可访问）
  ```
- **虚拟内存作为缓存**：对比第 6 章

  | 属性 | L1 缓存 | 虚拟内存（主存↔磁盘）|
  |------|---------|---------------------|
  | 缓存块大小 | 64 字节 | 页面 4KB |
  | 相联度 | 8-16 路组相联 | 全相联 |
  | 管理方式 | 硬件（自动）| 软件（OS）+ 硬件（MMU/TLB）|
  | 不命中代价 | ~200 周期 | ~10ms（数百万周期）|
  | 替换策略 | LRU（硬件近似）| 软件算法（LRU/Clock）|
  | 写策略 | 写回 | 写回（脏页延迟写回磁盘）|

## 一句话总结

> 数据是位（第 2 章），位被指令操作（第 3 章），懂了硬件才能写出快代码（第 5-6 章），散落的 `.o` 被链接成可执行文件（第 7 章），OS 用异常控制流管理它的运行（第 8 章），最终用虚拟内存给每个进程一个"整个世界都是我的"的幻觉（第 9 章）。
