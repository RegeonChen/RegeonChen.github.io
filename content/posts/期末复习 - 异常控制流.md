# 期末复习 — 异常控制流（Exceptional Control Flow）

> CSAPP 第 8 章 · 对应教材 §8.1–§8.8

***

## 一、概述

### 1.1 什么是 ECF

CPU 从开机到关机不断执行指令，这个指令序列就是**控制流（Control Flow）**。正常情况下控制流是平滑、可预测的：PC 顺序递增，或通过 `jmp`/`call`/`ret` 等指令在程序控制的范围内跳转。

当系统发生某些事件（硬件中断、系统调用、缺页等），控制流会发生程序**无法预见的突变**，这类突变称为**异常控制流（Exceptional Control Flow, ECF）**。

### 1.2 ECF 贯穿系统的四个部分

CSAPP 第 8 章将 ECF 分为四个递进的部分讲解，从硬件底层到应用层逐步展开：

|      部分     | 主题                    |    所在层次   | 典型例子                                        |
| :---------: | --------------------- | :-------: | ------------------------------------------- |
|  **ECF I**  | 异常（Exceptions）        |    硬件层    | 中断、缺页、除零、`syscall`                          |
|  **ECF II** | 进程（Processes）         |   操作系统层   | `fork()`、`execve()`、上下文切换                   |
| **ECF III** | 信号（Signals）           | 操作系统/用户接口 | `SIGINT`、`SIGSEGV`、`SIGCHLD`                |
|  **ECF IV** | 非本地跳转（Nonlocal Jumps） |    应用层    | `setjmp`/`longjmp`、`sigsetjmp`/`siglongjmp` |

> 从硬件到软件、从内核到用户，四个部分层层递进、逐步抽象。ECF 是理解计算机系统的关键——它将看似分离的"异常处理"、"进程调度"、"信号机制"统一在一个框架下。

## 二、异常（Exceptions）— 硬件层

### 2.1 异常表（Exception Table / IDT）

操作系统在启动时初始化一张**异常表**（x86-64 上称为 IDT，Interrupt Descriptor Table），CPU 通过 IDTR 寄存器指向它。发生异常时，CPU 硬件自动：

```
异常号 n
  → 从 IDTR 寄存器获取异常表基址
  → 访问异常表[n] 获取处理程序入口地址
  → 切换到内核栈，保存用户态上下文
  → 跳转到处理程序执行
```

### 2.2 四类异常

|         类别         | 触发原因         |  同步/异步 | 返回行为         | 典型案例                             |
| :----------------: | ------------ | :----: | ------------ | -------------------------------- |
| **中断 (Interrupt)** | 外部 I/O 设备信号  | **异步** | 返回到**下一条**指令 | 键盘输入、时钟中断、磁盘 I/O 完成              |
|    **陷阱 (Trap)**   | **有意**执行的指令  | **同步** | 返回到**下一条**指令 | `syscall`、`int 0x80`、`int 3`（断点） |
|   **故障 (Fault)**   | **可能可恢复**的错误 | **同步** | 返回到**当前**指令  | 缺页故障、COW 写时复制                    |
|   **终止 (Abort)**   | **不可恢复**的错误  | **同步** | **不返回**      | 硬件校验错误、非法操作码                     |

#### 同步 vs 异步

- **同步异常**：由当前正在执行的指令**直接导致**（陷阱、故障、终止）。可预测、可复现，不能被屏蔽。
- **异步异常**：来自**外部设备**，与当前指令无关（中断）。不可预测，可以通过 `cli`/`sti` 指令遮蔽。

#### 中断与 DMA 的协作：一次磁盘 I/O 的完整中断流程

中断最典型的场景是**磁盘 I/O 完成通知**，其中 DMA 承担了数据搬运的主角。以下是一次 `read()` 从磁盘读取数据时，中断 + DMA 的完整协作流程：

```
时间线 →

① 用户进程调用 read(fd, buf, n)
   → syscall 指令触发陷阱（Trap）
   → 内核 sys_read() 向磁盘控制器写入命令：
        "读取第 N 号扇区，DMA 目标地址 = 主存 0x..."
   → 磁盘控制器开始寻道 + 旋转等待
   → 内核将当前进程移出 CPU（等待 I/O），切换到其他进程运行

② DMA 传输阶段（CPU 完全不参与）
   磁盘磁头就位后，数据开始流入：
     磁盘 → 磁盘控制器 → 内存总线 → 主存（目的地 0x...）
   整个过程由磁盘控制器内部的 DMA 引擎驱动
   CPU 此时在跑别的进程，对此完全不知情

③ DMA 传输完成 → 触发中断（Interrupt）
   磁盘控制器通过中断控制器向 CPU 发出中断信号
   中断号 → 查 IDT → 找到磁盘中断处理程序

④ CPU 响应中断
   - 硬件自动保存当前进程的上下文（%rip、EFLAGS 等压入内核栈）
   - 切换到内核栈，跳转到磁盘中断处理程序
   - 中断处理程序：
       检查磁盘状态寄存器 → 确认传输成功
       标记等待该数据的进程为"就绪"（可以继续运行了）
       向中断控制器发 EOI（End of Interrupt）应答

⑤ 中断返回
   iret 指令 → 恢复之前被打断的进程的上下文
   （不一定是发起 read 的那个进程——调度器可能选择另一个就绪进程）
   若调度器选中原进程：read() 返回读取的字节数，进程继续执行
```

**涉及的 ECF 类型总结**：

| 步骤 | ECF 类型 | 触发方 | 作用 |
|:---:|:-------:|:-----:|------|
| ① | **陷阱（Trap）** | CPU 执行 `syscall` | 进入内核发起 I/O 命令 |
| ② | （无异常） | DMA 引擎 | 磁盘控制器直接搬运数据到主存，CPU 空闲 |
| ③–④ | **中断（Interrupt）** | 磁盘控制器 | 异步通知 CPU 数据已就绪 |
| ⑤ | `iret` 返回 | 内核 | 恢复被打断的用户态执行 |

> **关键启示**：一次磁盘读取只涉及两次 ECF——开头的陷阱和结尾的中断。中间的 DMA 传输阶段 CPU 完全"不知情"，这是系统高效运转的核心设计。没有 DMA，每条磁盘数据都要先经过 CPU 寄存器再到内存，CPU 会被慢速的磁盘完全拖死。

#### 关键语义：故障为什么返回"当前指令"而非"下一条"？

四种异常中，**故障**的返回行为最为特殊——它不像陷阱和中断那样返回到下一条指令，而是**返回到触发故障的那条指令本身，重新执行一次**。

理解这个设计，需要对比三类同步异常的"使用场景"：

**陷阱（Trap）返回下一条 — 因为"事情已经办完了"**

```
  syscall        ← 这条指令触发陷阱
  下一条指令      ← 内核已经把活干完了，回来继续走
```

`syscall` 是一条**完整的、有意的**请求。执行它之后，内核已经把系统调用处理完毕（比如 `read()` 已经把数据填入了缓冲区），程序自然应该继续执行下一条指令。

**故障（Fault）返回当前 — 因为"刚才的事没做成，修好了再来"**

```
  movl 0x7f000000, %eax   ← 这条指令触发缺页
  下一条指令               ← 不能跳到这里！
```

`movl 0x7f000000, %eax` 是一条**被中断的**指令——它试图从地址 `0x7f000000` 处读取 4 字节数据并加载到 `%eax`，但因为页不在物理内存中而**未能完成**。缺页处理程序的工作是"把缺失的页从磁盘加载到内存、更新页表"，但此时 `%eax` 中还没有被写入正确的值。

如果返回到下一条指令：程序会以为 `mov` 已经执行成功，拿着一个无意义的旧值继续运行，必然出错。

如果返回到当前指令：CPU 重新执行 `movl 0x7f000000, %eax`，这一次页表有效，MMU 成功转换地址，数据真正被加载到 `%eax` 中。**对程序来说，仿佛什么都没有发生过。** 这就是虚拟内存透明性的核心——缺页对用户程序完全不可见。

**终止（Abort）不返回 — 因为进程已直接终止**

```
  触发致命错误  → 进程直接终止，不存在"继续执行"的问题
```

### 2.3 系统调用（陷阱的典型实现）

**传统方式（x86 32-bit）：** `int 0x80`，通过触发第 128 号异常进入内核。

**现代方式（x86-64）：** `syscall` 指令，CPU 从 MSR\_LSTAR 寄存器获取内核入口地址，比 `int 0x80` 快得多（硬件优化路径）。

```
用户程序调用 read(fd, buf, 256)
  → glibc 封装函数
  → mov $0, %rax       ; 系统调用号 sys_read = 0
  → mov fd, %rdi       ; 参数
  → syscall            ; 触发陷阱
  → CPU 切换到 Ring 0，保存上下文
  → 内核 entry_SYSCALL_64
  → sys_call_table[0] → sys_read()
  → 内核处理完成，返回用户态
```

**异常表 vs 系统调用跳转表：**

|  对比  |    异常表（IDT）    | 系统调用跳转表（sys\_call\_table） |
| :--: | :------------: | :-----------------------: |
| 由谁使用 | CPU **硬件**自动查找 |        内核**软件**手动索引       |
| 覆盖范围 |   **所有**异常类型   |      **仅**系统调用（陷阱的一种）     |
| 索引依据 |    异常号（硬件决定）   |      系统调用号（`%rax` 指定）     |

- 异常表是"总机"，每个异常号分到不同处理程序
- 系统调用跳转表是"系统调用话务员"手里的"分机表"，按编号转接具体内核函数

## 三、进程（Processes）— 操作系统层

> **进程（Process）：** “一个执行中的程序的实例。”。

### 3.1 进程是 ECF 的产物

进程的三大抽象——独立的逻辑控制流、私有的地址空间、统一的 I/O 接口——全部依赖 ECF：

| 进程特性         | 依赖的 ECF 机制            |
| ------------ | --------------------- |
| 进程调度 / 上下文切换 | **时钟中断** + 系统调用陷阱     |
| 虚拟内存隔离       | **缺页故障** + **COW 故障** |
| 系统调用接口       | **陷阱**                |
| 进程终止清理       | **终止** + 信号           |

> **上下文切换（Context Switch）：** 当内核决定暂停当前进程、切换到另一个进程时，会保存当前进程的 CPU 寄存器、页表基址寄存器（CR3）、程序计数器（RIP）、register files 和 status registers 等状态快照，然后恢复目标进程之前保存的快照，使其从上次被中断处继续执行。整个过程由 ECF 驱动（时钟中断或系统调用），对进程本身完全透明。

### 3.2 进程的逻辑控制流

```
进程 A 看到的控制流（平滑连续）：
  ┌───┬───┬───┬───┬───┬───┐
  │A1 │A2 │A3 │A4 │A5 │A6 │
  └───┴───┴───┴───┴───┴───┘

CPU 上实际发生的（A 和 B 交替）：
  ┌───┬───┬───┬───┬───┬───┐
  │A1 │A2 │B1 │A3 │A4 │B2 │  ← 每次时钟中断都可能触发调度
  └───┴───┴───┴───┴───┴───┘
```

### 3.3 并发与并行

#### 并发（Concurrency）

**定义：** 多个进程的指令在**时间上重叠执行**——它们的生命周期有时间上的交集，但不一定同时执行。

```
单核 CPU 上的"并发"（时间片轮转）：
  ┌──┬──┬──┬──┬──┬──┬──┬──┬──┐
  │A1│A2│B1│A3│C1│B2│A4│C2│B3│  一个核心轮流执行 A、B、C
  └──┴──┴──┴──┴──┴──┴──┴──┴──┘
   ←──────── 时间 →────────
   A、B、C 三个进程在"同一段时间内"都有进展 → 并发
```

**进程的正常执行由时钟中断打断→上下文切换→调度另一个进程运行。"并发"是 ECF 的直接产物——没有时钟中断就没有上下文切换，就没有并发。**

#### 并行（Parallelism）

**定义：** 多个进程的指令在**同一时刻**真正同时执行在**不同 CPU 核心**上。

```
双核 CPU 上的"并行"：
  核心 0: ┌──┬──┬──┬──┬──┬──┐
          │A1│A2│A3│A4│A5│A6│  A 独占核心 0
          └──┴──┴──┴──┴──┴──┘
  核心 1: ┌──┬──┬──┬──┬──┬──┐
          │B1│B2│B3│B4│B5│B6│  B 独占核心 1
          └──┴──┴──┴──┴──┴──┘
                    时间 →
  在时刻 T：A3 和 B3 同时执行 → 并行
```

**并行是硬件能力的体现**——它要求系统有多个 CPU 核心（或超线程），每个核心独立执行指令流。

#### 并发 vs 并行：关键区别

| 对比 | 并发 (Concurrency) | 并行 (Parallelism) |
|:---:|:-----------------:|:-----------------:|
| **含义** | 多个任务在**同一段时间**内都有进展 | 多个任务在**同一时刻**同时执行 |
| **硬件要求** | 单核即可（通过时间片轮转） | 必须多核或多处理器 |
| **本质** | **逻辑上**的交错执行 | **物理上**的同时执行 |
| **与 ECF 的关系** | 直接依赖时钟中断和上下文切换 | 依赖多核硬件，ECF 提供调度基础 |

```
并发 + 并行的常见组合（多核 + 时间片）：
  核心 0: ┌──┬──┬──┬──┬──┬──┬──┐
          │A1│C1│A2│C2│A3│C3│A4│  在核心 0 上，A 和 C 并发（交替执行）
          └──┴──┴──┴──┴──┴──┴──┘
  核心 1: ┌──┬──┬──┬──┬──┬──┬──┐
          │B1│B2│B3│B4│B5│B6│B7│  在核心 1 上，B 独立执行
          └──┴──┴──┴──┴──┴──┴──┘
                    时间 →
  A 和 B 是并行的（不同核心同时执行）
  A 和 C 是并发的（同一核心交替执行）
```

**实际系统中的典型场景：**

- 单核 CPU 上运行浏览器 + 编辑器 → **并发**（时间片切换），不并行
- 4 核 CPU 上运行 2 个进程 → **并行**（各自占用一个核心）
- 4 核 CPU 上运行 10 个进程 → **既有并发也有并行**（部分进程并行运行，部分进程分时复用）

### 3.4 进程三态模型与 ECF 的对应

CSAPP 将进程分为三种状态：

```
         ┌─────────────────────────────────┐
         │         运行 (Running)           │
         │   进程正在 CPU 上执行，或          │
         │   等待被调度（包括因 I/O 暂时让出  │
         │   CPU 的情况）                   │
         └────┬──────────────────┬─────────┘
              │                  │
       收到 SIGSTOP/      进程终止
       SIGTSTP 信号       (exit 或收到
              │          致命信号)
              ▼                  │
    ┌─────────────────┐          ▼
    │  暂停 (Stopped)  │  ┌──────────────┐
    │ 被信号挂起，      │  │ 终止 (Terminated)│
    │ 直到收到 SIGCONT │  │ 永久结束       │
    └────────┬────────┘  └──────────────┘
             │
       收到 SIGCONT 信号
             │
             └──→ 回到 Running
```

| 状态 | 含义 | 典型场景 |
|:---:|------|---------|
| **Running** | 正在执行或可被调度执行（包括因 I/O 等待而暂时让出 CPU 的情况） | 进程正常运行中 |
| **Stopped** | 被信号挂起，暂停执行，直到收到 `SIGCONT` 才能恢复 | `Ctrl+Z` 发送 `SIGTSTP`；调试器中的断点暂停 |
| **Terminated** | 永久终止，不再存在 | 调用 `exit()`、`main` 返回、收到 `SIGKILL` 等致命信号 |

> **注意：** CSAPP 的 "Running" 是一个宽泛的概念，涵盖了传统 OS 教材中"运行"和"就绪"两种状态。进程中因 `read()` 等待磁盘而被暂时移出 CPU，在 CSAPP 视角下仍属于 Running 态——因为它只是暂时让出 CPU，最终会被内核唤醒并继续执行。

所有状态转换均由 ECF 驱动：

- **Running → Stopped**：进程收到 `SIGSTOP`/`SIGTSTP` 信号
- **Stopped → Running**：进程收到 `SIGCONT` 信号
- **Running → Terminated**：进程调用 `exit()`、`main` 返回、收到 `SIGKILL`/`SIGSEGV` 等致命信号（前两者通过**陷阱**进入内核，后两者属于**信号**机制）

#### 实例：一次 `read()` 调用中的控制流切换

以下场景展示进程 A 调用 `read(fd, buf, 256)` 读取磁盘文件时，进程 A、进程 B 以及内核之间如何通过 ECF 协作（注意：A 和 B 在整个过程中都处于 **Running** 态）：

```
时间线 →

① 进程 A 在 CPU 上执行到 read()
   A: n = read(fd, buf, 256);
        ↓
   glibc 封装 → mov $0, %rax; syscall
        ↓
   陷阱（Trap）→ CPU 切换到内核态
        ↓
   内核 sys_read() 向磁盘控制器发起 DMA 请求
   内核发现需等待 I/O → A 暂时让出 CPU（仍属于 Running 态）
   调度器选中进程 B 投入运行

② 上下文切换到进程 B，B 在 CPU 上执行
   B: for (int i = 0; i < 1000000; i++);
        ↓
   ...（某个时刻）磁盘 DMA 传输完成
   磁盘控制器向 CPU 发送中断信号
        ↓
   中断（Interrupt）→ CPU 暂停 B，切换到内核态
        ↓
   内核中断处理程序：数据已就绪 → A 可以继续执行了

③ 内核返回前，调度器做决策：
   - 如果 A 优先级更高 → 上下文切换到 A
   - 如果 B 时间片未耗尽 → 继续运行 B

④ 调度器选中 A
   上下文切换：保存 B 的快照，恢复 A 的快照
   内核将磁盘数据拷贝到用户缓冲区 buf
   sysret 返回用户态，read() 返回读取的字节数
   A: printf("read %d bytes\n", n);  // 继续执行
```

>  **DMA（Direct Memory Access，直接内存访问）** 是一种允许外部设备（如磁盘、网卡、显卡）与计算机内存直接进行数据交换的技术，它完全绕过了中央处理器（CPU）的干预。 

**本例中涉及的 ECF 类型：**

| 步骤 | ECF 类型 | 作用 |
|:---:|:-------:|------|
| ① A 执行 `syscall` | **陷阱（Trap）** | 进入内核发起 I/O |
| ② 磁盘完成信号 | **中断（Interrupt）** | 通知内核 I/O 完成 |
| ④ 回到 A | **上下文切换** | 恢复进程 A 的执行 |

### 3.5 关键系统调用

`fork()`、`execve()`、`waitpid()` 是进程管理的三个核心系统调用，它们通常配合使用：

```
父进程                           子进程
  │                               │
  ├─ pid = fork() ───────┐        │
  │  （返回子进程 PID）    │        │
  │                      └──────→ ├─ pid = fork()
  │                               │  （返回 0）
  │                               ├─ execve("/bin/ls", ...)
  │                               │  替换为 ls 程序
  │                               ├─ ls 执行完毕，exit()
  │                               │
  ├─ waitpid(pid, ...) ←─────────┘  子进程终止，内核通知父进程
  │  收割子进程，回收资源
  ▼
继续执行
```

- **`fork()`**：创建进程。调用一次，返回两次——父进程得到子进程 PID，子进程得到 `0`。二者通过不同返回值走向不同的代码分支。
- **`execve()`**：替换程序。子进程通常立即调用 `execve()` 加载新程序，把从父进程复制来的地址空间替换为目标程序。
- **`waitpid()`**：回收资源。父进程调用它等待子进程结束，获取退出状态，防止子进程成为僵尸进程。

> 在 Shell 中每输入一条命令（如 `ls -l`），Shell 就执行一次 **fork + execve + waitpid** 的完整流程。

#### `fork()` — 调用一次，返回两次

```c
#include <unistd.h>
pid_t fork(void);
// 返回值：
//   > 0  → 这是父进程，返回值 = 子进程的 PID（进程 ID）
//   = 0  → 这是子进程
//   < 0  → fork 失败（如系统进程数达上限），不会创建子进程
```

**关键语义**：

| 返回值 | 意味着 | 典型代码分支 |
|:-----:|--------|:----------|
| **`> 0`** | 当前在**父进程**中执行，返回值是内核分配的**子进程 PID** | `if (pid > 0) { /* parent code */ }` |
| **`= 0`** | 当前在**子进程**中执行 | `if (pid == 0) { /* child code */ }` |
| **`< 0`** | `fork` 调用失败（系统资源不足），没有创建任何子进程 | 错误处理路径，不应该继续执行父子代码 |

```c
#include "csapp.h"

int main()
{
    pid_t pid;
    int x = 1;

    pid = Fork();
    if (pid == 0) { /* child */
        printf("child : x=%d\n", ++x);
        exit(0);
    }

    /* parent */
    printf("parent: x=%d\n", --x);
    exit(0);
}
```

**程序分析：**

1. **`x` 的值为何不同？** `fork()` 之后，子进程获得父进程地址空间的独立副本（页表克隆，所有可写页标记为 COW 只读）。子进程执行 `++x` 触发 COW 缺页，获得私有副本后修改为 `2`；父进程执行 `--x` 修改自己的 `x` 为 `0`。**父子进程的 `x` 互不影响——它们各自拥有一份独立的数据。**

2. **输出顺序不确定：** 父子进程并发执行，调度器决定谁先运行。两种可能的输出：

   ```
   可能输出 A:                   可能输出 B:
   parent: x=0                  child : x=2
   child : x=2                  parent: x=0
   ```

   > 两者都是正确的——`fork()` 后父子执行顺序无保证。在 CSAPP 课本中，"parent: x=0 / child : x=2" 的顺序被称为"一个典型输出"。

3. **`exit(0)` 的作用：** 正常终止进程。若子进程还要运行其他程序，则会调用 `execve()` 替换自身。

#### `execve()` — 替换进程镜像

```c
int execve(const char *filename, char *const argv[], char *const envp[]);
//  成功：不返回   失败：返回 -1
```

`execve` 是 Linux 中唯一真正加载并执行程序的系统调用，其他变体（`execl`、`execv`、`execlp` 等）都是它的包装。

```c
// 以 ls -l /tmp 为例：
char *argv[] = {"ls", "-l", "/tmp", NULL};
char *envp[] = {"PATH=/bin:/usr/bin", NULL};

if (Fork() == 0) {                           // 子进程
    execve("/bin/ls", argv, envp);           // 替换为 ls
    perror("execve failed");                 // 执行到这里说明 execve 失败
    exit(1);
}
```

**工作原理（内核步骤）：**

1. 从磁盘加载 `/bin/ls` 的 ELF 文件头
2. 检查执行权限
3. 释放当前进程的地址空间（代码段、数据段、堆、栈全部清空）
4. 按 ELF segment 表将 `.text`、`.data`、`.bss` 等映射到虚拟内存
5. 将 `argv[]`、`envp[]` 写入新栈底，设置 `argc`
6. PC 跳转到 ELF entry point → 开始执行 `ls` 的 `main(argc, argv, envp)`

```
execve 前后对比（以 ls -l /tmp 为例）：

  执行前（Shell 子进程）                执行后（已变成 ls）
  ┌──────────────────┐                ┌──────────────────┐
  │ Shell 代码/数据    │               │ ls 代码 (.text)   │
  │ fork 出的副本      │     替换       ├──────────────────┤
  ├──────────────────┤   ───────→     │ ls 数据 (.data)   │
  │ 堆（副本）         │                ├──────────────────┤
  ├──────────────────┤                │ 堆（重置）         │
  │ 栈（副本）         │                ├──────────────────┤
  └──────────────────┘                │ 新栈（含 argv/    │
                                      │   envp 字符串）   │
  PID 不变 ✓                          └──────────────────┘
  PCB 不变 ✓                          PID 不变 ✓
  打开的文件描述符保留 ✓               文件描述符保留 ✓
                                      但 pc → ls 的 _start
```

**关键理解：**

- **只换内容，不换 ID**：进程仍是同一个（PID 和 PCB 不变），但地址空间被完全替换。就像火车头不变，车厢全换了。
- **文件描述符默认保留**：`stdin/stdout/stderr` 不会关闭。这正是 Shell 重定向（`ls > out.txt`）的原理：子进程 `fork` 后先 `dup2` 修改 fd，再 `execve`，新程序继承重定向后的 fd。
- **调用成功不返回**：一旦 `execve` 成功，当前程序的代码已被覆盖，根本没有"返回"的机会。只有失败时才返回 `-1`。

#### `waitpid()` — 收割子进程

等待子进程终止并回收其资源。若父进程不调用 `wait`，已终止的子进程将永远停留在**僵尸状态**，占用内核资源。

```c
#include <sys/wait.h>
pid_t waitpid(pid_t pid, int *status, int options);
// 返回值：
//   > 0  → 终止的子进程 PID
//   = 0  → (仅当 options=WNOHANG 时) 子进程尚未终止，需轮询
//   < 0  → 错误（如没有该子进程、被信号中断等）
```

| 参数/返回值 | 含义 | 常见用法 |
|:----------:|------|---------|
| `pid > 0` | 等待 PID 等于该值的**特定子进程** | `waitpid(child_pid, &status, 0)` |
| `pid = -1` | 等待**任意子进程**（谁先终止收割谁）| `waitpid(-1, &status, 0)` |
| `options = 0` | **阻塞等待**直到子进程终止 | 默认行为 |
| `options = WNOHANG` | **非阻塞**：若子进程未终止则立即返回 0 | `while (waitpid(-1, NULL, WNOHANG) > 0)` 批量收割 |
| `options = WUNTRACED` | 同时报告**已停止**的子进程（收到 SIGSTOP）| 调试器使用 |
| **返回值 `> 0`** | 成功收割，值为**终止的子进程 PID** | 判断是哪个子进程结束了 |
| **返回值 `= 0`** | 仅 WNOHANG 模式下：没有子进程已终止，还需等待 | 循环轮询时的退出条件 |
| **返回值 `< 0`** | 错误（已无子进程、信号中断等）| 检查 `errno` |

**检查子进程退出状态**（通过 `status` 参数）：

```c
int status;
pid_t wpid = waitpid(child_pid, &status, 0);

if (wpid > 0) {                                     // 成功收割
    if (WIFEXITED(status))                          // 子进程正常终止
        printf("exit code: %d\n", WEXITSTATUS(status));
    else if (WIFSIGNALED(status))                   // 子进程被信号杀死
        printf("killed by signal: %d\n", WTERMSIG(status));
    else if (WIFSTOPPED(status))                    // 子进程被暂停
        printf("stopped by signal: %d\n", WSTOPSIG(status));
}
```

| 宏 | 用途 | 何时为真 |
|:--|------|---------|
| `WIFEXITED(status)` | 检查是否**正常 exit()** | 子进程 `exit(n)` 或 `main` return |
| `WEXITSTATUS(status)` | 提取**退出码**（0–255）| `WIFEXITED` 为真时 |
| `WIFSIGNALED(status)` | 检查是否**被信号杀死** | 收到 SIGKILL/SIGSEGV 等致命信号 |
| `WTERMSIG(status)` | 提取**杀死它的信号编号** | `WIFSIGNALED` 为真时 |
| `WIFSTOPPED(status)` | 检查是否**被挂起** | 收到 SIGSTOP/SIGTSTP 等暂停信号 |
| `WSTOPSIG(status)` | 提取**挂起它的信号编号** | `WIFSTOPPED` 为真时 |

#### 僵尸进程与回收机制

当子进程通过 `exit()` 终止时，其用户态资源（代码段、数据段、堆、栈）被释放，但内核仍然保留该子进程的 **PCB（进程控制块）**——此时子进程进入**僵尸状态**（Zombie）。PCB 中保存着子进程的退出状态码，供父进程通过 `waitpid()` 读取。

```
子进程 exit() → 用户资源释放 → PCB 保留（僵尸状态）→ 父进程 waitpid() → 内核回收 PCB → 进程彻底消亡
```

**为什么会有僵尸进程？** 子进程终止后，父进程可能需要知道子进程的退出原因（正常结束/收到信号），所以内核不能立刻销毁 PCB，必须等父进程主动来"收割"。

**孤儿进程的处理：** 如果父进程在子进程之前终止，子进程就变成"孤儿"。内核会将孤儿进程的父进程重新指派给 `init` 进程（PID = 1），由 `init` 负责在孤儿终止时调用 `waitpid` 将其回收——因此**孤儿进程不会变成僵尸**。

**僵尸进程的危害与防范：**

- 大量僵尸进程堆积会耗尽 PID 和内核资源，导致无法创建新进程。
- 防范方式：父进程必须及时调用 `waitpid()`；或者通过**信号处理机制**（详见后续章节）在子进程终止时自动收割，避免漏收。

## 四、信号（Signals）— 用户态软件异常

### 4.1 概念

信号是操作系统向进程发送的**通知**，告知某种事件已发生。可以把信号理解为"进程级别的异常"：中断是硬件通知 CPU，信号是内核通知进程。

### 4.2 常见信号

|     信号    |  值  |   默认行为  | 触发场景                   |
| :-------: | :-: | :-----: | ---------------------- |
|  `SIGINT` |  2  |    终止   | `Ctrl+C`               |
| `SIGKILL` |  9  |    终止   | `kill -9`（**不可捕获/忽略**） |
| `SIGSEGV` |  11 | 终止+core | 段错误（非法内存访问）            |
| `SIGPIPE` |  13 |    终止   | 向已关闭的管道写数据             |
| `SIGALRM` |  14 |    终止   | `alarm()` 定时器到期        |
| `SIGTERM` |  15 |    终止   | `kill` 默认信号            |
| `SIGCHLD` |  17 |    忽略   | 子进程终止/暂停               |
| `SIGSTOP` |  19 |    暂停   | `Ctrl+Z`（**不可捕获/忽略**）  |

### 4.3 信号生命周期

```
发送 (Send) → 待处理 (Pending) → 递送 (Delivery)
```

- **发送**：内核或进程调用 `kill()`，在目标进程的 pending 位图中置位。
- **待处理**：信号已发送但尚未处理。**同一信号不排队**——pending 位图只记录"有/无"，不计次数。
- **阻塞**：进程可以阻塞（Block）某些信号。阻塞 ≠ 忽略；阻塞只是延迟递送，解除阻塞后 pending 信号会被递送。
- **递送**：处理信号——执行默认动作（终止/暂停/忽略）或**捕获**（跳转到用户自定义的信号处理程序）。

### 4.4 信号处理程序的三个难点

**① 待处理信号不排队**
同一信号在 pending 期间多次到达，只处理一次。因此 SIGCHLD 处理程序中必须用循环收割（否则一个 handler 调用只能收割一个子进程）：

```c
void sigchld_handler(int sig) {
    int old_errno = errno;        // 保存 errno，防止被 waitpid 覆盖
    // WNOHANG 非阻塞轮询：检查所有已终止但未收割的子进程
    while (waitpid(-1, NULL, WNOHANG) > 0)
        ;
    errno = old_errno;            // 恢复 errno
}
```

循环终止条件 `waitpid() > 0` 意味着：
- 返回 `> 0` → 成功收割一个子进程，继续循环收割下一个
- 返回 `= 0` → WNOHANG 模式下没有更多已终止的子进程，退出循环
- 返回 `< 0`（= -1）→ 已无子进程，退出循环（不是错误）

**② 可重入性（Reentrancy）**
信号处理程序中**只能调用 async-signal-safe 函数**。`printf`、`malloc`、`exit` 等因为内部使用全局锁或共享数据结构，在信号处理程序中不安全。

安全函数示例：`write()`、`_exit()`、`kill()`、`waitpid()`、`sigprocmask()`。

**③ 并发访问全局变量**
主程序和信号处理程序可能并发修改全局变量。用 `volatile sig_atomic_t` 声明，且只做单次读写（不能用 `+=` 等读-改-写操作，因为不是原子操作）。

### 4.5 信号相关的关键 API 与返回值

#### `kill()` — 发送信号

```c
#include <signal.h>
int kill(pid_t pid, int sig);
// 返回值：成功 = 0，失败 = -1（如进程不存在、无权限）
```

| pid 参数 | 含义 |
|:-------:|------|
| `pid > 0` | 将信号 `sig` 发给 PID = pid 的**特定进程** |
| `pid = 0` | 发给**本进程组**的所有进程 |
| `pid < 0` | 发给 PGID = \|pid\| 的**整个进程组** |

#### `sigprocmask()` — 阻塞/解除阻塞信号

```c
int sigprocmask(int how, const sigset_t *set, sigset_t *oldset);
// 返回值：成功 = 0，失败 = -1
```

| how 参数 | 含义 |
|:-------:|------|
| `SIG_BLOCK` | 将 `set` 中的信号加入阻塞集合（原来阻塞的继续阻塞）|
| `SIG_UNBLOCK` | 将 `set` 中的信号从阻塞集合中移除 |
| `SIG_SETMASK` | 直接用 `set` 替换当前阻塞集合 |

#### `sigaction()` — 注册信号处理程序

```c
int sigaction(int signum, const struct sigaction *act, struct sigaction *oldact);
// 返回值：成功 = 0，失败 = -1
```

#### 注册 SIGCHLD 处理程序的完整示例

```c
#include <signal.h>
#include <sys/wait.h>

void sigchld_handler(int sig) {
    int old_errno = errno;
    while (waitpid(-1, NULL, WNOHANG) > 0)   // 循环收割所有已终止子进程
        ;
    errno = old_errno;
}

int main() {
    struct sigaction sa;
    sigemptyset(&sa.sa_mask);      // 初始化信号掩码为空
    sa.sa_flags = 0;
    sa.sa_handler = sigchld_handler;

    if (sigaction(SIGCHLD, &sa, NULL) < 0) {  // < 0 表示注册失败
        perror("sigaction");
        exit(1);
    }

    // 此后 fork 出的子进程终止时，内核自动调用 sigchld_handler
    ...
}
```

### 4.6 信号处理程序执行流程

```
主程序执行中...
  → 信号到达
  → 内核检查 disposition = 捕获
  → 内核保存主程序上下文到用户栈
  → 跳转到 handler (Ring 3)
  → handler 执行完成，return
  → 通过 __NR_rt_sigreturn 系统调用进入内核
  → 内核恢复主程序上下文
  → 继续执行主程序（从被中断处）
```

## 五、非本地跳转（Nonlocal Jumps）— 用户态控制流绕过

### 5.1 `setjmp` / `longjmp`

允许程序跨越多个函数栈帧直接跳转：

```c
#include <setjmp.h>
int setjmp(jmp_buf env);          // 初次调用返回 0；longjmp 后"返回"非 0
void longjmp(jmp_buf env, int val); // 跳回对应的 setjmp 位置，使 setjmp 返回 val
```

| 函数 | 返回值语义 |
|:---:|----------|
| `setjmp(env)` | **初次调用**：返回 `0`，并将当前栈环境保存到 `env` |
| `setjmp(env)` | **`longjmp` 跳回时**：返回 `longjmp(env, val)` 中传入的 `val`（`val` 不能为 0，若传入 0 则 setjmp 返回 1）|
| `longjmp(env, val)` | **不返回**——执行流直接跳转到 `setjmp` 处 |

```c
jmp_buf env;

int main() {
    if (setjmp(env) == 0) {         // 第一次 setjmp: 返回 0 → 进入 if
        func1();                    // 正常执行
    } else {                        // longjmp 后 setjmp 返回非 0 → 跳到这里
        printf("Recovered!\n");
    }
}

void func1() { func2(); }
void func2() { longjmp(env, 1); }   // longjmp 不返回，直接跳回 main
```

**关键限制：** `longjmp` 只能跳回**尚未返回**的 `setjmp` 所在函数。如果该函数已经 `return` 了，则行为未定义。

### 5.2 在信号处理中的应用

```c
sigjmp_buf env;  // 保存/恢复信号屏蔽字

void handler(int sig) {
    siglongjmp(env, 1);
}

int main() {
    signal(SIGINT, handler);
    if (sigsetjmp(env, 1) == 0) {
        while (1) sleep(1);       // 主循环
    } else {
        printf("Ctrl+C caught, exiting\n");
    }
}
```

典型应用：Shell 的 `Ctrl+C` 中断恢复、深度嵌套的错误处理。

## 七、复习要点
