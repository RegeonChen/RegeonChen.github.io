## 写一个合适的文档

和很多人的直觉可能不一样，即便 AI Agent 已经迭代到了现在这个程度，代码的可读性以及文档化对 Agent 来说仍然很重要，某种意义上可能比对人类还重要。具体体现在：

- 大语言模型本质上是语言理解与生成的工具，在 Coding Agent 的工作场景中，正确理解复杂的上下文是成功的基础，而**代码本身携带的有价值信息**（良好的命名规则、模块划分、函数定义、注释等等）越多，越能帮助模型生成正确的代码；而误导性的成分越多，越容易造成模型的误解与错判。
- 如果是人类与 Agent 共同维护的代码，我们可以通过一些**特殊约定的标签**来限制和规范 Agent 行为，比如通过特殊的注释标注保护不可更改的代码片段。
- 以人为主的工程通常追求团队的稳定，但我们经常更换 Agent 工具， Agent 也不具备人类那样的长期记忆与稳定风格，所以更需要通过**标准化代码风格和专门的约束文档**来弥补。

与建立专用文档与 Agent 互动相比，在代码中适用良好的风格，配合精心设计的协作标签，能更经济的使用上下文（因为 *Agent* 通常只访问与任务有关的片段），也符合人类维护代码的习惯。

但不管怎么说，上文所述大部分都是写代码与维护代码中所需的了。人类用 AI Agent 开发项目的第一步，依然是写一个合适的文档。

下面我将讨论，如何通过若干个 md 文档的引导，用 AI Agent 开发一个项目：

### Step 1. `init.md`

由人类起草。

含有该产品的目标定位 + 核心关注（比如和其他市场竞品差异化的功能） + 约束

### Step 2. 与AI讨论完善

完善功能规划 + 确认技术选型和方案

### Step 3. `Agents.md`

- 由 AI 生成。每次 AI 响应需求时都需要读一遍这一个文档。

- 全局约束（也可以在某个子目录下进行约束）
- 固化关键决策与约束的记忆体。

不是针对特定的细节和功能，而是记录诸如："如何进行功能测试和验收"、"如何回答我的询问"等 你希望 AI Agent 遵循的工作流。

以下是一个可参照的 `Agents.md` 可含有的内容示例：

- Project Standards & Constraints
  - Project Goal：一句话目标
  - Architecture：架构原则：解耦、协议、引擎支持
  - Tech Stack：语言、工具、框架
  - Key Features：功能清单
  - Coding Conventions：编码约定
  - Current Status：UI /Engine / Config / Protocol 现状
  - Design Decisions：规则管理、棋盘参数化等关键决策
  - Recent Notes：近期实现细节和注意事项（如：通过对话纠正的错误，应记录到本文档中）
  - Roadmap：未来规划
  - Known Issues：已知问题和暂时解决方案

### Step 4. `Plan.md`

可以让多个模型（甚至是多重人机交叉检验）围绕同一个 `Plan.md` 进行修改和完善，一步步确定开发计划执行的每一个部分。

> 为什么要对一个项目分出不同的阶段？
>
> 核心原则：每个项目都有可以进行单独验证的标准

以下是一个可参照的 `Plan.md` 结构示例：

- Project Plan
  - Phases：产品的关键里程碑，较大的可验收阶段
  - Sub－phases：某些里程碑可能包含2-3个子里程碑
  - Tasks：里程碑/子里程碑包含的具体任务
    - Overall Goal：每个任务的核心目标
    - Task Detail：任务具体内容，包括实现方案及：
    - Affected Files：受影响的文件（含新建）
    - Key Design：关键设计如数据结构、核心模块及算法等
    - Verification：任务完成的可验证标准，包括自动和人工验证

复杂的项目可以将计划分为 `PLAN.md` 和 `/phases` 目录下的具体每个 phase 的详细计划。
