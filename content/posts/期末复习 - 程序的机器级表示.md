## 一、概述

### 1.1 本章在全书中的位置

第 3 章回答一个核心问题：**我们写的 C 代码，在 CPU 眼中到底是什么？** 它是连接"高级语言"和"硬件"的桥梁。

```
C 源代码                  汇编代码                 机器码
int sum(int a, int b)    sum:                    ...
{                         movl %edi, %eax
    return a + b;         addl %esi, %eax
}                         ret
     ↑                      ↑                      ↑
  第1章导论              第3章汇编               第7章链接
                         (本章核心)           (这些机器码进入 .text 段)
```

**前驱后继关系**：

| 章节 | 与第 3 章的关系 |
|:----:|---------------|
| 第 2 章（← 依赖）| 数据类型大小决定 `movb/w/l/q` 的选择；立即数的补码/十六进制表示 |
| 第 5 章（→ 输出）| 汇编让你能"看到"优化的效果：几条指令变成一条？循环展开省了多少条？|
| 第 7 章（→ 输出）| `.o` 文件中的重定位表描述的就是汇编指令中地址占位符 |
| 第 8 章（→ 输出）| `syscall` 指令触发陷阱；栈帧布局为 `setjmp/longjmp` 提供基础 |
| 第 9 章（→ 输出）| 每次 `movq (%rax), %rbx` 都要经过地址翻译（MMU/TLB/页表）|

## 二、基础概念（§3.1–§3.3）

### 2.1 三层架构：OS、ISA、微架构

理解汇编之前，需要先区分三个层次：

| 特性 | OS | ISA（指令集架构）| Microarchitecture（微架构）|
|:---:|:--:|:-------------:|:-----------------------:|
| 本质 | 软件 | 抽象规范（接口）| 硬件实现 |
| 可见对象 | 应用程序员、最终用户 | 编译器作者、汇编程序员 | 芯片设计工程师 |
| 变化频率 | 高（频繁更新版本）| 低（保持软件兼容，变化缓慢）| 中（每代新产品都会更新）|
| 主要目标 | 资源管理、易用性、安全性 | 兼容性、编程模型定义 | 性能（IPC）、能效比、面积优化 |
| 举例 | Linux / Windows / macOS | **x86-64** / ARM / RISC-V | Intel Core i7-13700K / Apple M2 / AMD Zen 4 |

> 本章站在 **ISA 层**：定义 CPU 提供哪些指令、哪些寄存器、什么寻址模式、ABI 如何约定——这些对汇编程序员/编译器作者可见。

### 2.2 核心概念：PC、寄存器、条件码

| 术语 | 全称 | 本质 | 主要功能 | 类比 |
|:---:|------|------|---------|------|
| **PC** | Program Counter | 专用寄存器 | 指向下一条指令的地址，控制执行流 | 读书时的手指 |
| **Register** | Register | 高速存储单元 | 暂存数据和中间结果，供快速运算 | 桌上的草稿纸 |
| **Condition Codes** | Condition Codes / Flags | 状态位集合 | 记录上一次运算的结果特征（零、负、溢出等）| 运算后的记号 |

### 2.3 汇编语法风格

x86-64 汇编有两种主流语法，CSAPP 使用 **AT&T 语法**（GCC/objdump 默认），区别于 Intel 语法：

| 特征 | AT&T（本书使用）| Intel |
|------|:-------------:|:----:|
| 源/目的操作数顺序 | `op src, dst` | `op dst, src` |
| 寄存器前缀 | `%rax` | `rax` |
| 立即数前缀 | `$42` | `42` |
| 内存寻址 | `disp(%base, %index, scale)` | `[base+index*scale+disp]` |
| 指令后缀 | `movq`（含大小后缀）| `mov`（操作数大小决定）|

## 三、访问信息 — 寄存器、操作数、数据传送（§3.4）

### 3.1 x86-64 寄存器全集

x86-64 有 16 个 64 位通用寄存器，可访问低 32/16/8 位子集：

| 64 位 | 32 位 | 16 位 | 8 位 | 特殊用途 / 记忆要点 |
|:-----:|:-----:|:-----:|:----:|-------------------|
| `%rax` | `%eax` | `%ax` | `%al` | **返回值** |
| `%rbx` | `%ebx` | `%bx` | `%bl` | **被调用者保存** |
| `%rcx` | `%ecx` | `%cx` | `%cl` | 第 4 个参数 |
| `%rdx` | `%edx` | `%dx` | `%dl` | 第 3 个参数 |
| `%rsi` | `%esi` | `%si` | `%sil` | 第 2 个参数 |
| `%rdi` | `%edi` | `%di` | `%dil` | **第 1 个参数** |
| `%rbp` | `%ebp` | `%bp` | `%bpl` | **被调用者保存**、帧指针（可选）|
| `%rsp` | `%esp` | `%sp` | `%spl` | **栈指针** |
| `%r8` | `%r8d` | `%r8w` | `%r8b` | 第 5 个参数 |
| `%r9` | `%r9d` | `%r9w` | `%r9b` | 第 6 个参数 |
| `%r10`–`%r11` | | | | 调用者保存 |
| `%r12`–`%r15` | | | | **被调用者保存** |

> **必记口诀**：返回值 `%rax`，参数 `%rdi %rsi %rdx %rcx %r8 %r9`（共 6 个寄存器传参），栈指针 `%rsp`。

### 3.2 操作数寻址模式（核心中的核心）

汇编指令的操作数可以来自三种位置，加上 AT&T 语法的特定写法：

| 类型 | 格式（AT&T）| 操作数值 | 名称 | 示例 |
|:---:|-----------|---------|------|------|
| 立即数 | `$Imm` | `Imm` | 立即数寻址 | `$42`, `$-231`, `$0x400` |
| 寄存器 | `%r_a` | `R[%r_a]` | 寄存器寻址 | `%rax` |
| 内存（直接）| `Imm` | `M[Imm]` | 绝对寻址 | `0x400` |
| 内存（间接）| `(%r_a)` | `M[R[%r_a]]` | 间接寻址 | `(%rax)` |
| 内存（基址+偏移）| `Imm(%r_b)` | `M[Imm+R[%r_b]]` | 基址+偏移 | `8(%rbp)` |
| 内存（完整）| `Imm(%r_b, %r_i, s)` | `M[Imm+R[%r_b]+R[%r_i]·s]` | 比例变址 | `8(%rdi, %rsi, 4)` |

**通用公式**：`Imm(%rb, %ri, s)` = **`M[Imm + R[%rb] + R[%ri] × s]`**，其中比例因子 $s \in \{1, 2, 4, 8\}$。

> **`R[]` 与 `M[]` 的含义**：`R[%rax]` 表示读取 CPU 内部的寄存器 `%rax` 中的值（极快，0 周期延迟）；`M[addr]` 表示以 `addr` 为地址去内存（DRAM）中读取数据（较慢，需经过缓存层级）。带括号的汇编语法（如 `(%rax)`）本质上就是在做一次 `M[]` 索引——先取 `R[%rax]` 得到地址，再用该地址访存。

**AT&T 语法关键约定**：
- `%` 开头表示**寄存器**，例如 `%rax` 表示寄存器本身的内容
- 寄存器外加括号表示"以该寄存器中的数值作为**内存地址**，去该地址处读取或写入的数据"，如 `(%rax)` 表示 `M[R[%rax]]`
- `$` 开头表示**直接的数值**（立即数），如 `$0x400`、`$-231`

#### `%rip`-相对寻址

x86-64 引入了一种关键的寻址方式——以 `%rip`（当前指令地址）为基准、加偏移来定位目标：

**格式**：`Imm(%rip)`  → `M[Imm + R[%rip]]`

```asm
movq global(%rip), %rax    # 读取全局变量 global，地址 = %rip + 偏移
call printf@PLT            # 内部也是 call *offset(%rip)
```

**与绝对寻址的对比**：

| 寻址方式 | 格式 | 计算方式 | 特点 |
|:------:|------|---------|------|
| **绝对寻址** | `0x400000` | 地址 = `0x400000`（固定常量）| 硬编码，位置不可变 |
| **`%rip`-相对** | `global(%rip)` | 地址 = `当前%rip + 偏移` | 位置无关，代码可搬到任意地址 |

**为什么重要**：`%rip`-相对寻址是实现 PIC（Position-Independent Code，第 7 章）的基石。由于地址是基于"距离"而非"固定位置"，同一段代码可以被加载到任意虚拟地址而不需要修改机器码中的地址。这使得：
- 动态共享库（`.so`）可以被不同进程映射到不同的虚拟地址
- ASLR（第 8/9 章）可以每次将代码段随机放置
- 多个进程可以共享同一份 `.so` 的物理页，但各自映射到不同虚拟地址

> **与第 7 章的直接对应**：`.o` 文件中的重定位条目 `R_X86_64_PC32` 正是描述这类 `%rip`-相对引用——链接器只需计算"目标地址 − `%rip` 当前值"填入偏移即可，无需知道绝对地址。

### 3.3 数据传送指令 MOV

#### 3.3.1 `movq` 基本格式与操作数类型

在 x86-64 架构中，数据移动是最基本的操作之一。

**基本指令格式**：`movq Source, Dest`

- **含义**：将源操作数（Source）的数据**复制**到目的操作数（Dest），源不变。
- **后缀 `q`**：代表 "Quad word"，表示操作的数据长度为 **8 字节（64 位）**。同理 `l`=4 字节，`w`=2 字节，`b`=1 字节。
- **AT&T 语法注意**：源在左，目的在右（即 `Source → Dest`），与 Intel 语法的 `Dest, Source` 顺序相反。

操作数可以是以下三种类型之一：

| 类型 | 描述 | 示例 | 备注 |
|:---|------|:---|------|
| **立即数 (Immediate)** | 常量整数数据 | `$0x400`, `$-533` | 类似 C 常量，加 `$` 前缀。通常编码为 1/2/4 字节 |
| **寄存器 (Register)** | CPU 内部 16 个通用整数寄存器 | `%rax`, `%r13` | `%rsp` 通常保留用于栈管理 |
| **内存 (Memory)** | 以寄存器中的值作为地址访问内存 | `(%rax)` | 括号即解引用，可搭配 §3.2 的全部寻址模式 |

#### 3.3.2 `movq` 的五种合法组合

**x86-64 不允许单条指令直接从内存复制到内存**——必须通过寄存器中转。

| 源 → 目的 | 汇编示例 | C 语言类比 | 说明 |
|:--------:|:--------|:---------|------|
| 立即数 → 寄存器 | `movq $0x4, %rax` | `temp = 0x4;` | 常数加载到寄存器 |
| 立即数 → 内存 | `movq $-147, (%rax)` | `*p = -147;` | 常数直接写入内存 |
| 寄存器 → 寄存器 | `movq %rax, %rdx` | `temp2 = temp1;` | 寄存器间复制 |
| 寄存器 → 内存 | `movq %rax, (%rdx)` | `*p = temp;` | 寄存器值存入内存 |
| 内存 → 寄存器 | `movq (%rax), %rdx` | `temp = *p;` | 从内存读取到寄存器 |

> ⚠️ **非法组合**：`movq (%rax), (%rdx)` — 内存到内存！必须拆为两步：`movq (%rax), %r10` → `movq %r10, (%rdx)`。

#### 3.3.3 MOV 指令大小变体

| 指令 | 操作数大小 | 功能 |
|:---:|:--------:|------|
| `movb` | 1 字节 | 传送字节 |
| `movw` | 2 字节 | 传送字 |
| `movl` | 4 字节 | 传送双字 |
| `movq` | 8 字节 | 传送四字 |
| `movabsq` | 8 字节 | 传送绝对四字（立即数 → 寄存器，仅用于 64 位立即数）|

**限制**：`movq` 只能以 32 位补码数作为立即数，需要 64 位立即数时用 `movabsq`。

#### 零扩展传送（源 < 目标宽度）

| 指令 | 源大小 | 目标大小 | 功能 |
|:---:|:-----:|:------:|------|
| `movzbw` | 1B | 2B | 零扩展 字节→字 |
| `movzbl` | 1B | 4B | 零扩展 字节→双字 |
| `movzbq` | 1B | 8B | 零扩展 字节→四字 |
| `movzwl` | 2B | 4B | 零扩展 字→双字 |
| `movzwq` | 2B | 8B | 零扩展 字→四字 |

**特殊规律**：`movl` 写入 32 位目标寄存器时，会自动将高 32 位**清零**（相当于自带零扩展）。因此没有 `movzlq` 这种指令。

#### 符号扩展传送

| 指令 | 源大小 | 目标大小 | 功能 |
|:---:|:-----:|:------:|------|
| `movsbw` | 1B | 2B | 符号扩展 字节→字 |
| `movsbl` | 1B | 4B | 符号扩展 字节→双字 |
| `movsbq` | 1B | 8B | 符号扩展 字节→四字 |
| `movswl` | 2B | 4B | 符号扩展 字→双字 |
| `movswq` | 2B | 8B | 符号扩展 字→四字 |
| `movslq` | 4B | 8B | 符号扩展 双字→四字 |
| `cltq` | — | — | `%eax` 符号扩展到 `%rax` |

> **与第 2 章的对应**：`movz` 系列 = 零扩展（第 2 章 §2.2.6），`movs` 系列 = 符号扩展（第 2 章 §2.2.5）。

### 3.4 栈操作：pushq 与 popq

```
         高地址
    |     ...     |
    +-------------+
    |             |  ← 已有的栈数据
    +-------------+
    | 新 push 的值 |  ← rsp 减小后指向这里
    +-------------+  ← 原 rsp
    |     ...     |
         低地址
```

| 指令 | 等价操作 | 效果 |
|:---:|---------|------|
| `pushq S` | `sub $8, %rsp`; `movq S, (%rsp)` | 先减少 `%rsp`，再写入 |
| `popq D` | `movq (%rsp), D`; `add $8, %rsp` | 先读出，再增加 `%rsp` |

**栈的成长方向**：从高地址向低地址。`pushq` 时 `%rsp` **减少** 8；`popq` 时 `%rsp` **增加** 8。

> **存储顺序**：大端小端别搞混——栈上存一个 `int` 类型值 `0x12345678`，低地址存 `78`，高地址存 `12`（小端序，第 2 章）。

### 3.5 综合示例：数据传送 + 过程调用

以下是一个 C 函数和其对应的汇编代码，综合展示了 `pushq`、`popq`、`movq`、`call` 等核心指令如何协作完成一次带指针参数的函数调用：

```c
// 声明一个外部函数 plus，接收两个 long 类型参数并返回一个 long 类型值
long plus(long x, long y);

// 定义 sumstore 函数
void sumstore(long x, long y, long *dest)
{
    // 1. 调用 plus(x, y)，并将结果存入局部变量 t
    long t = plus(x, y);

    // 2. 将 t 的值写入 dest 指向的内存地址
    *dest = t;
}
```

```asm
sumstore:           # sumstore 函数的入口标签
        pushq   %rbx      # 1. 保存 %rbx 寄存器的值到栈上
        movq    %rdx, %rbx# 2. 将 %rdx (即第三个参数 dest) 复制到 %rbx
        call    plus      # 3. 调用 plus 函数
        movq    %rax, (%rbx)# 4. 将 %rax (plus的返回值) 存入 %rbx 指向的内存地址
        popq    %rbx      # 5. 从栈上恢复 %rbx 寄存器的原始值
        ret               # 6. 从 sumstore 函数返回
```

**逐行分析——本节知识点一一对应**：

| 步骤 | 指令 | 涉及的知识点 | 发生了什么 |
|:---:|------|:----------:|-----------|
| 1 | `pushq %rbx` | 栈操作、被调用者保存 | `%rbx` 是**被调用者保存**寄存器，`sumstore` 要使用它，因此必须先把旧值保存在栈上。`%rsp` 减 8，`%rbx` 旧值写入栈顶。 |
| 2 | `movq %rdx, %rbx` | 数据传送、寄存器、参数传递 | `%rdx` 是第 3 个参数（即 `dest` 指针）。因为接下来要调用 `plus`，`call` 可能破坏 `%rdx`（调用者保存），所以把 `dest` 搬家到被调用者保存的 `%rbx` 中——**这是跨函数调用时保护数据的经典手法**。 |
| 3 | `call plus` | `call` 指令、参数传递 | `call` 自动把返回地址压栈，然后跳转到 `plus`。传入 `plus` 的参数 `x` 和 `y` 已经在 `%rdi` 和 `%rsi` 中——`sumstore` 自己没有改过它们，直接透传。 |
| 4 | `movq %rax, (%rbx)` | 内存间接寻址、返回值 | `plus` 的返回值在 `%rax` 中。`(%rbx)` 是**内存间接寻址**——以 `%rbx` 中的值（即 `dest` 指针）为地址，写入 `*dest = t`。 |
| 5 | `popq %rbx` | 栈操作、被调用者保存 | 从栈顶恢复 `%rbx` 的原始值（与步骤 1 对称），`%rsp` 加 8。 |
| 6 | `ret` | `ret` 指令 | 从栈顶弹出返回地址到 `%rip`，回到调用者继续执行。 |

**关键设计决策 —— 为什么用 `%rbx` 而不是其他寄存器？**

`sumstore` 拿到 `dest` 指针（在 `%rdx` 中）后，需要把它保留到 `call plus` 返回之后。但 `%rdx` 是**调用者保存**寄存器——`plus` 函数有权覆盖它。因此 `sumstore` 把 `dest` 搬到 `%rbx`（被调用者保存），并在函数开头/结尾用 `pushq %rbx` / `popq %rbx` 配对保护。这就是 ABI 约定在实际代码中的体现。

> **一句话理解**：必须用 `%rbx` 的根本原因是 `call plus` 这条指令——`%rdx` 属于调用者保存寄存器（`call` 之后不可靠，——ABI 明确规定，被调用函数 plus 有权利随意修改 %rdx 以及 %rdi 、 %rsi 、 %rcx 、 %r8 – %r11 、 %rax），而 `%rbx` 属于被调用者保存寄存器（ABI 保证任何函数返回后 `%rbx` 值与调用前一致）。相当于把 `dest` 从"战区"（调用者保存寄存器）搬进"防空洞"（被调用者保存寄存器），再用 `pushq`/`popq` 把防空洞里原有的东西暂存到栈上。

## 四、算术与逻辑操作（§3.5）

### 4.1 加载有效地址 lea

`leaq S, D`：**不访存**，仅计算地址 `S` 并存入 `D`。

```
leaq 8(%rdi, %rsi, 4), %rax
// 效果：%rax = %rdi + %rsi * 4 + 8
// 注意：不会去读取 M[%rdi + %rsi*4 + 8]！
```

`lea` 的妙用：在不使用乘法指令 `imul` 的情况下，利用比例变址计算某些常数乘法。但比例因子 $s \in \{1, 2, 4, 8\}$ （分别对应 00/01/10/11），因此**单条 `lea` 只能计算 `x * (1/2/3/4/5/8/9) + b` 这类组合**。

以 `x * 12`（假设 `x` 在 `%rdi` 中）为例，展示正确的推理过程：

**思路**：$12 = 3 \times 4$，分两步走——`lea` 算 ×3，再 `sal`（左移）算 ×4：

```asm
leaq (%rdi, %rdi, 2), %rax   # %rax = x + x*2 = x*3
salq $2, %rax                # %rax = (x*3) << 2 = x*12
```

> **结论**：单条 `lea` 的能力受限于比例因子集合 {1, 2, 4, 8} 和最多两个求和项（基址 + 变址×比例）。$x \times 12$ 无法用单条 `lea` 完成，最简翻译是 `lea (x, x, 2)` 得到 `x*3`，再接一条 `sal $2` 左移 2 位完成 ×4。

## 五、控制 — 条件码、跳转与循环（§3.6）

### 5.1 条件码

CPU 维护一组单 bit 的条件码寄存器，由最近的算术/逻辑指令自动设置：

| 条件码 | 全称 | 含义 | 何时为 1 |
|:-----:|------|------|---------|
| **CF** | Carry Flag | 进位标志 | 无符号运算产生进位（加法）或借位（减法）|
| **ZF** | Zero Flag | 零标志 | 结果为 0 |
| **SF** | Sign Flag | 符号标志 | 结果为负数（最高位 = 1）|
| **OF** | Overflow Flag | 溢出标志 | 有符号运算产生溢出（补码溢出）|

> `leaq` **不改变**任何条件码（这是 lea 和算术指令的关键区别）。

### 5.2 比较与测试指令

| 指令 | 行为 | 用途 |
|:---:|------|------|
| `cmp S1, S2` | `S2 - S1`（仅设置条件码，不存结果）| 比较两个数 |
| `test S1, S2` | `S1 & S2`（仅设置条件码，不存结果）| 检查某位是否为 0、是否为 NULL |

**典型 idiom**：
```c
testq %rax, %rax   // 等价于检查 %rax == 0（设置 ZF）
cmpq  $0, %rdi      // 等价于检查参数 == 0
```

### 5.3 使用条件码：set 指令

条件码（CF/ZF/SF/OF 等）保存在一个专用的标志寄存器 `EFLAGS`/`RFLAGS` 中，**汇编程序不能直接读写它们**——没有 `movq %zf, %rax` 这种操作。要"读取"条件码，必须通过一组间接指令。

#### set 指令族

`set` 指令根据条件码的组合，将一个单字节寄存器（如 `%al`）设置为 0 或 1：

| 指令 | 同义名 | 效果 | 条件 |
|:---:|:-----:|------|------|
| `sete` | `setz` | `D = ZF` | 相等 / 为零 |
| `setne` | `setnz` | `D = ~ZF` | 不等 / 非零 |
| `sets` | | `D = SF` | 负数 |
| `setg` | `setnle` | `D = ~(SF^OF) & ~ZF` | 有符号 大于 |
| `setge` | `setnl` | `D = ~(SF^OF)` | 有符号 大于等于 |
| `setl` | `setnge` | `D = SF^OF` | 有符号 小于 |
| `setle` | `setng` | `D = (SF^OF) \| ZF` | 有符号 小于等于 |
| `seta` | `setnbe` | `D = ~CF & ~ZF` | 无符号 大于 |
| `setb` | `setnae` | `D = CF` | 无符号 小于 |

**关键限制**：`set` 指令的**目标只能是一个单字节寄存器**（`%al`, `%bl`, `%cl`, `%dl` 等低 8 位寄存器）。这意味 `setg %rax` 是非法的——必须写成 `setg %al`。

#### 将 set 的结果扩展到 64 位

`setg %al` 的结果只写入了 `%rax` 的最低字节，高 56 位保留了原来的值（可能是垃圾）。要得到一个干净的 `0` 或 `1` 放在完整的 64 位寄存器中，需要两步配合：

```asm
cmpq   %rsi, %rdi       # 比较 a 和 b，设置条件码
setg   %al              # 如果 a > b（有符号），%al = 1；否则 %al = 0
movzbl %al, %eax        # movzbl零扩展：%eax = 低字节零扩展到 32 位
                        # 同时，x86-64 写入 32 位寄存器会自动将高 32 位清零
                        # 因此整个 %rax = 0x0000000000000000 或 0x0000000000000001
```

**这里涉及两重 x86-64 特殊机制**：

1. **`movzbl` 的零扩展**：`movzbl %al, %eax` 将 `%al`（单字节）零扩展到 `%eax`（32 位），高 24 位补 0。此时低 32 位（`%eax`）是干净的 0 或 1。
2. **写入 32 位寄存器的自动清零**：在所有 x86-64 工作模式下，任何写入 32 位寄存器（如 `%eax`、`%ebx`）的指令都会**自动将对应的 64 位寄存器的高 32 位清零**。因此 `movzbl %al, %eax` 执行后，不仅低 32 位是 0 或 1，高 32 位也被硬件自动清零——整个 `%rax` 就是干净的 `0x0000000000000000` 或 `0x0000000000000001`。

> **一句话理解**：`cmp` 设条件码 → `setg %al` 读条件码到单字节 → `movzbl %al, %eax` 把单字节零扩展 + 高 32 位自动清零 = 得到完整的 64 位布尔值。这就是 C 语言 `a > b` 这个表达式返回 `0` 或 `1` 在汇编层面的完整实现。

### 5.4 条件跳转指令

| 指令 | 同义名 | 跳转条件 | 基于的条件码 |
|:---:|:-----:|---------|:---------:|
| `je` | `jz` | 相等 / 为零 | `ZF` |
| `jne` | `jnz` | 不等 / 非零 | `~ZF` |
| `js` | | 负数 | `SF` |
| `jns` | | 非负 | `~SF` |
| `jg` | `jnle` | 有符号 大于 | `~(SF^OF) & ~ZF` |
| `jge` | `jnl` | 有符号 大于等于 | `~(SF^OF)` |
| `jl` | `jnge` | 有符号 小于 | `SF^OF` |
| `jle` | `jng` | 有符号 小于等于 | `(SF^OF) \| ZF` |
| `ja` | `jnbe` | 无符号 大于 | `~CF & ~ZF` |
| `jae` | `jnb` | 无符号 大于等于 | `~CF` |
| `jb` | `jnae` | 无符号 小于 | `CF` |
| `jbe` | `jna` | 无符号 小于等于 | `CF \| ZF` |

**记忆技巧**：有符号用 "g/l"（greater/less），无符号用 "a/b"（above/below）。

### 5.5 条件传送 cmov

在**流水线处理器**中，条件跳转会导致**分支预测失败**的惩罚。条件传送 `cmov` 通过**同时计算两个分支的结果、最后按条件选择**，避免了跳转：

```
// 条件跳转版本（有分支）            // 条件传送版本（无分支）
cmpq %rsi, %rdi                  cmpq %rsi, %rdi
jge  .L1                         cmovge %rdx, %rax
movq %rdx, %rax                  // 条件码满足时把 %rdx 拷贝到 %rax
ret                              ret
.L1:
movq %rcx, %rax
ret
```

#### cmov 的代价与三大限制

条件传送的核心机制是"**无论条件是否满足，两个分支的值都必须先被计算出来**，再按条件二选一"——这种"先算后选"导致以下三类场景下不能使用或不应使用 `cmov`：

##### 1. 昂贵的计算 (Expensive Computations) — 导致性能更差

```c
val = Test(x) ? Hard1(x) : Hard2(x);
```

如果 `Hard1` 和 `Hard2` 是非常耗时的函数（复杂数学运算、长循环等），`cmov` 意味着 CPU 必须**同时执行两个函数**，最后只取其中一个结果、丢弃另一个。这比传统的 `if-else`（只执行一个分支就跳走）慢得多。

> **结论**：只有当两个分支的计算都非常简单（如单条加法或寄存器搬移）时，`cmov` 才划算。

##### 2. 危险的计算 (Risky Computations) — 导致程序崩溃

```c
val = p ? *p : 0;        // 空指针检查
```

这是经典的空指针保护逻辑。在 `if-else` 中，`p` 为空时会跳过 `*p`（解引用），程序安全。但在 `cmov` 中，CPU 会预先尝试计算 `*p` 的值——如果 `p == NULL`，这条预计算就直接触发**段错误**（SIGSEGV），程序崩溃。

##### 3. 带副作用的计算 (Computations with Side Effects) — 导致语义错误

```c
val = x > 0 ? x *= 7 : x += 3;
```

两个分支都会修改 `x`（这就是"副作用"）。在 `if-else` 中只有一个分支被执行，`x` 只被修改一次。但在 `cmov` 逻辑中，CPU 可能**同时执行** `x*=7` 和 `x+=3`，导致 `x` 被意外修改两次，结果不可预测。

> **总原则**：`cmov` 只能用于**纯数据选择**——两个分支都是无副作用的简单表达式求值，且求值本身不会引发安全问题（如空指针解引用）。

### 5.6 循环的三种翻译方式

C 语言的 `while`/`for` 循环在汇编层面有三种翻译方式：

| 方式 | 结构 | 特点 |
|:---:|------|------|
| **跳转到中间** | `goto test` 跳到末尾再判断 | `while` 的最直接翻译 |
| **guarded-do** | 先判断一次，再 `do-while` | `for` 常用；入口先检查，内部是 do-while 循环 |
| **do-while** | 先执行再判断 | 如果编译器能证明至少执行一次 |

```
// 跳转到中间翻译：
    goto test
loop:
    body
test:
    if (condition) goto loop

// guarded-do 翻译：
    if (!condition) goto done
loop:
    body
    if (condition) goto loop
done:
```

### 5.7 switch 语句与跳转表

`switch` 语句在 case 数量较多、case 值分布密集时，编译器不会生成一系列 `cmp + je` 逐条比较（$O(n)$），而是用**跳转表（Jump Table）** 实现 $O(1)$ 分支。

#### 编译器生成跳转表的条件

并非所有 `switch` 都会被编译为跳转表：

| 条件 | 说明 |
|:---:|------|
| case 值**密集**（如 100–106）| 密集 → 跳转表紧凑高效；稀疏 → 跳转表浪费空间 |
| case 数量 **≥ 4~5 个** | 太少不值得建表 |
| case 值可映射为连续索引 | 通过 `index = val - min` 映射 |

不满足以上条件时，编译器会退化为 `if-else` 链（二分查找）或混合策略。

#### 跳转表的工作原理

假设 C 代码如下：

```c
int switch_eg(int x) {
    int result;
    switch (x) {
        case 100: result = x + 1; break;
        case 101: result = x * 2; break;
        case 102: result = x - 3; break;
        case 103: result = x << 2; break;
        default:  result = 0;   break;
    }
    return result;
}
```

编译器将其翻译为以下汇编逻辑：

```asm
# 假设 x 在 %rdi 中
switch_eg:
        leaq    100(%rdi), %rax      # 计算 index = x - 100
        cmpq    $3, %rax              # 检查 index 是否 > 3（因为 100~103 共 4 个 case）
        ja      .L8                   # 如果 index > 3（无符号比较），跳转到 default
        jmp     *.L4(, %rax, 8)       # 间接跳转！以 index 为索引查跳转表

.L3:    # case 100
        leaq    1(%rdi), %rax         # result = x + 1
        ret
.L5:    # case 101
        leaq    (%rdi, %rdi), %rax    # result = x * 2
        ret
.L6:    # case 102
        leaq    -3(%rdi), %rax         # result = x - 3
        ret
.L7:    # case 103
        salq    $2, %rdi
        movq    %rdi, %rax            # result = x << 2
        ret
.L8:    # default
        xorq    %rax, %rax            # result = 0
        ret

# 跳转表（放在 .rodata 段中）：
.L4:
        .quad   .L3    # index=0 → case 100 的代码地址
        .quad   .L5    # index=1 → case 101 的代码地址
        .quad   .L6    # index=2 → case 102 的代码地址
        .quad   .L7    # index=3 → case 103 的代码地址
```

#### 逐步分析

**第 1 步：构建索引**

```asm
leaq 100(%rdi), %rax          # %rax = x - 100
                               # 注意：这是用 lea 做减法，因为 lea 不算术，不设条件码
```

将 case 值平移到以 0 为起点：`x = 100 → index = 0`，`x = 103 → index = 3`。

**第 2 步：范围检查**

```asm
cmpq $3, %rax                  # 比较 index 与最大值 3
ja   .L8                       # 如果 index > 3（无符号），说明不在 100~103 范围内
```

这里用的是**无符号比较** `ja`，因为 `index` 被当作无符号数：如果 `x < 100`（如 `x=99`），`index = -1`，在无符号解释下是一个巨大值（远大于 3），条件成立，跳转到 `default`。这就是经典的双重检查技巧——**一个无符号比较同时捕获"小于 100"和"大于 103"两种越界情况**。

**第 3 步：间接跳转**

```asm
jmp *.L4(, %rax, 8)            # 跳转到 M[.L4 + index * 8]
```

`*` 表示间接跳转——目标地址不是直接写在指令中，而是从跳转表中读取。`.L4` 是跳转表的基址（在 `.rodata` 段中），每个条目 8 字节（`.quad`），以 `index` 为下标索引。

```
跳转表内存布局：
.L4:     .quad .L3    ← index=0 读取这里，得到 .L3 的地址
.L4+8:   .quad .L5    ← index=1
.L4+16:  .quad .L6    ← index=2
.L4+24:  .quad .L7    ← index=3
```

> **跳转表的核心价值**：无论有多少个 case，查找时间都是 $O(1)$——一次范围检查 + 一次间接跳转。而 `if-else` 链在最坏情况下需要比较所有 case（$O(n)$）。

#### 经典例题：从汇编 + 跳转表还原 C 代码（CSAPP 3.62）

以下是一道典型的课后习题：给定汇编代码和跳转表，要求还原出完整的 C 代码。这和 Attack Lab / Bomb Lab 中"在反汇编中识别 switch 结构"是同一技能。

**已知汇编代码**（`n` 在 `%rdi` 中）：

```asm
potion:
        movq    %rdi, %rax           # 将原始 n 保存到 %rax
        subq    $60, %rdi             # %rdi = n - 60
        cmpq    $5, %rdi              # 比较 (n-60) 与 5
        ja      .L2                   # 如果 (n-60) > 5（无符号），跳到 default
        jmp     *.L4(, %rdi, 8)       # 间接跳转：以 (n-60) 为索引查跳转表

.L3:                                  # case 60:
        leaq    0(,%rax,8), %rax      #   result = 8 * n
        ret
.L5:                                  # case 62:
        leaq    (%rax,%rax,4), %rax   #   result = 5 * n
        ret
.L6:                                  # case 64:
        leaq    (%rax,%rax,2), %rax   #   result = 3 * n
        ret
.L7:                                  # case 65:
        salq    $4, %rax              #   result = n * 16
        ret
.L2:                                  # default (case 61, 63 及越界):
        xorq    %rax, %rax            #   result = 0
        ret
```

**已知跳转表**（`.rodata` 段中，8 字节 per entry）：

```
.L4:
        .quad   .L3    # index = 0
        .quad   .L2    # index = 1
        .quad   .L5    # index = 2
        .quad   .L2    # index = 3
        .quad   .L6    # index = 4
        .quad   .L7    # index = 5
```

**逐步分析**：

**第 1 步：定 min（最小 case 值）**

```asm
subq $60, %rdi       # %rdi = n - 60
```

`subq $60` 意味着 `index = n - 60`，即 `n = 60 + index`。因此最小 case 值 = **60**（对应 index = 0）。

**第 2 步：定范围（有多少个 case）**

```asm
cmpq $5, %rdi        # 比较 index 与 5
ja   .L2             # index > 5 → default
```

`cmpq $5` 意味着索引范围是 `[0, 5]`，共 **6 个表项**。因此 case 值覆盖 **60 ~ 65**（60 + 0 到 60 + 5）。

同时，`ja`（无符号比较）一次性捕获两种越界：`n < 60` 时 `index` 为负数（无符号解释下是巨大值，> 5）和 `n > 65` 时 `index > 5`。

**第 3 步：逐表项对应（核心）**

将跳转表的 6 个表项逐一映射到代码块：

| index | n 值 | 表项地址 | 指向标签 | 代码块的运算 | C 含义 |
|:-----:|:---:|:------:|:-------:|:-----------|:------:|
| 0 | 60 | `.L3` | `.L3` | `result = 8 * n` | `case 60: result = 8*n; break;` |
| 1 | 61 | `.L2` | `.L2` | `result = 0` | 同 default |
| 2 | 62 | `.L5` | `.L5` | `result = 5 * n` | `case 62: result = 5*n; break;` |
| 3 | 63 | `.L2` | `.L2` | `result = 0` | 同 default |
| 4 | 64 | `.L6` | `.L6` | `result = 3 * n` | `case 64: result = 3*n; break;` |
| 5 | 65 | `.L7` | `.L7` | `result = n * 16` | `case 65: result = n*16; break;` |

**关键发现**：n = 61 和 n = 63 虽然在跳转表范围内（不是越界跳 default），但它们的表项**直接指向 `.L2`（即 default 的代码块）**。所以这两个 case 的语义就是"执行 default"。在最终的 C 代码中，不需要显式写出 `case 61:` 和 `case 63:`——C 语言的 switch 只要 `default:` 在，任何未明确列出的 case 值都会自动落入 default。

**还原出的完整 C 代码**：

```c
long potion(long n) {
    long result;
    switch (n) {
        case 60:
            result = 8 * n;
            break;
        case 62:
            result = 5 * n;
            break;
        case 64:
            result = 3 * n;
            break;
        case 65:
            result = n << 4;      // n * 16, salq $4 对应左移 4 位
            break;
        default:                  // 覆盖 n=61, 63 以及所有越界值
            result = 0;
            break;
    }
    return result;
}
```

> **应试要点**：遇到这类题目，只需三步——① 从 `subq` 的偏移反推 min；② 从 `cmpq` 的参数反推表项数；③ 逐表项找对应代码块和 C 语义。跳转表项指向 default 代码块意味着对应的 case 值不需要显式写出来。

## 六、过程 — 栈帧、调用约定（§3.7）

### 6.1 栈帧（Stack Frame）

每个过程调用在栈上分配一个栈帧：

```
       高地址
  +----------------------+
  |    调用者的栈帧        |
  |    第 7 个参数及以后    |  ← 超过 6 个寄存器传参的放在这里
  |    返回地址            |  ← call 指令自动压入
  +----------------------+
  |    被保存的寄存器       |  ← 被调用者保存寄存器（如 %rbp, %rbx, %r12-%r15）
  |    局部变量            |  ← 寄存器不够用时溢出到这里
  |    临时空间            |
  +----------------------+  ← %rsp 指向栈顶
       低地址
```

### 6.2 call 与 ret

| 指令 | 效果 |
|:---:|------|
| `call Label` | `pushq %rip`（下一条指令地址）; `jmp Label` |
| `ret` | `popq %rip` |

**`call` 两步走**：把返回地址压栈 + 跳转。
**`ret` 一步走**：弹出返回地址到 `%rip`。

### 6.3 参数传递约定（x86-64 ABI）

| 参数位置 | 寄存器 |
|:------:|:-----:|
| 第 1 个 | `%rdi` |
| 第 2 个 | `%rsi` |
| 第 3 个 | `%rdx` |
| 第 4 个 | `%rcx` |
| 第 5 个 | `%r8` |
| 第 6 个 | `%r9` |
| 第 7+ 个 | **栈上**（从右向左压栈，调用者负责清理）|

返回值：`%rax`（64 位），返回结构体时可能用到 `%rdx` 做第二返回值。

### 6.4 寄存器保存约定

| 类型 | 寄存器 | 规则 |
|:---:|:------:|------|
| **调用者保存** | `%rax, %rcx, %rdx, %rsi, %rdi, %r8-%r11` | 调用者如需保留值，需在 `call` 前自己保存 |
| **被调用者保存** | `%rbx, %rbp, %r12-%r15` | 被调用者如果要使用，**必须**先保存到栈上，返回前恢复 |

> **记忆法**：参数寄存器（%rdi %rsi %rdx %rcx %r8 %r9）+ 返回值（%rax）都是调用者保存——因为它们本来就是"传话"用的，用完即弃。

### 6.5 过程的完整调用流程

```
调用者侧:                       被调用者侧:
  前 6 个参数 → 寄存器              push %rbp（保存旧帧指针）
  第 7+ 参数 → 压栈                movq %rsp, %rbp（设置新帧指针）
  call func                       如需使用被调用者保存寄存器 → push 到栈
      ↓                           分配局部变量空间（sub $N, %rsp）
  (返回地址被 call 自动压栈)       ... 函数体 ...
      ↓                           恢复被调用者保存寄存器（pop）
  call 返回后继续执行              movq %rbp, %rsp; pop %rbp
  (如有栈上传参，恢复 %rsp)        ret
```

## 七、数组分配与访问（§3.8）

### 7.1 数组定标寻址

数组元素地址 = 基址 + 索引 × 元素大小。这就是比例变址寻址的典型应用：

```
// int a[10];  a[i] 的地址:
movslq %esi, %rsi          // 索引 i 扩展到 64 位
movl   (%rdi, %rsi, 4), %eax   // M[%rdi + %rsi*4]
// 比例因子 s=4 因为 sizeof(int)=4
```

| C 类型 | sizeof | 比例因子 s |
|:------:|:------:|:--------:|
| `char` | 1 | 1 |
| `short`| 2 | 2 |
| `int` / `float` | 4 | 4 |
| `long` / `double` / 指针 | 8 | 8 |

### 7.2 嵌套数组

`int A[R][C]`，元素 `A[i][j]` 的地址：

$$\&A[i][j] = x_A + (i \cdot C + j) \cdot L$$

其中 $L$ 为元素大小（`sizeof(int) = 4`）。

```c
// A[i][j] 的汇编：
// 假设 %rdi = A 基址, %rsi = i, %rdx = j, C = 5
imulq $5, %rsi, %rax       // i * C = i*5
addq  %rdx, %rax           // i*C + j
movl  (%rdi, %rax, 4), %eax // M[A + (i*C+j)*4]
```

### 7.3 指针与数组

```
// 数组名 a 是指针常量（不可被赋值）
int a[10];
a = a + 1;   // 错误！a 不是左值

// 函数传参时数组退化为指针
void func(int a[])  ≡  void func(int *a)
```

## 八、异质数据结构 — struct 与 union（§3.9）

### 8.1 结构体 struct

```c
struct S {
    char  c;   // 偏移 0
    int   i;   // 偏移 4（而非 1 — 对齐！）
    short s;   // 偏移 8
};
// sizeof(struct S) = 12（末尾填充到 4 的倍数）
```

访问结构体成员时，编译器计算偏移：

```
movsbl (%rdi), %eax          // c (偏移 0)
movl   4(%rdi), %edx         // i (偏移 4)
movswl 8(%rdi), %ecx         // s (偏移 8)
```

### 8.2 联合体 union

union 的所有成员**共享同一块内存**，大小为最大成员的大小：

```c
union U {
    char c;    // 1 字节
    int  i;    // 4 字节
    long l;    // 8 字节
};
// sizeof(union U) = 8
```

**用法**：类型双关（type punning）；节省内存（同一位置不同时间存不同类型数据）。

## 九、对齐 Alignment（§3.10）

每种类型的地址必须是其大小的整数倍：

| 类型 | 对齐要求 |
|:---:|:------:|
| `char` | 1 字节对齐 |
| `short` | 2 字节对齐 |
| `int` / `float` | 4 字节对齐 |
| `long` / `double` / 指针 | 8 字节对齐 |

结构体的整体大小是其最大对齐要求的整数倍。编译器会在成员之间和末尾插入**填充字节**：

```c
struct S2 {
    int  i;   // 偏移 0
    char c;   // 偏移 4
    short s;  // 偏移 6（注意不是 5——对齐到 2）
};
// sizeof(struct S2) = 8
// 内存布局: | i[4] | c[1] | pad[1] | s[2] |
```

#### 栈帧对齐

对齐不仅影响结构体，还约束了**栈帧**的布局。x86-64 ABI 要求**在 `call` 指令执行前，`%rsp` 必须是 16 字节对齐的**（即 `%rsp % 16 == 0`）。

这意味着：
- `call` 自动压入 8 字节返回地址后，被调用函数看到的 `%rsp` 是 **8 mod 16**（即 16 字节对齐被破坏了一个 word）
- 被调用函数如果需要再 `call` 其他函数，就必须在调整 `%rsp`（分配局部变量空间）时保证 `%rsp + 8` 是 16 的倍数

```asm
sumstore:
        pushq   %rbx            # %rsp 减 8，恢复 16 字节对齐
        subq    $16, %rsp       # 再分配 16 字节局部空间（仍然是 16 对齐）
        ...
        addq    $16, %rsp       # 回收局部空间
        popq    %rbx            # 恢复 %rbx
        ret
```

**典型现象**：编译器经常在栈帧中插入无用的**额外 `subq $8, %rsp`**（或分配 8 的奇数倍的局部空间），唯一目的就是凑齐 16 字节对齐。这是阅读编译器生成的汇编代码时常见的"困惑点"——看到奇怪的栈偏移往往是因为对齐约束。

## 十、结合控制与数据（§3.11）

### 10.1 指针与内存

指针就是地址。`*p` 在汇编层面就是一次**内存访问**（用 `(%reg)` 间接寻址）。空指针解引用 → `movq $0, %rax; movq (%rax)` → 访问地址 0 → **段错误（SIGSEGV）**。

### 10.2 缓冲区溢出

当写入数据超过缓冲区大小时，会覆盖相邻的栈数据——最危险的是覆盖**返回地址**：

```
         调用者的栈
  +------------------------+
  | 返回地址 ←─── 被覆盖！  |  ← gets(buf) 写入过长字符串
  +------------------------+
  |    buf[8]               |  ← 从这里开始写，向下溢出  ↑↑↑
  +------------------------+
```

**防护机制**（第 3 章介绍，第 8 章深入）：

| 机制 | 原理 |
|:---:|------|
| **栈金丝雀（Stack Canary）** | 在返回地址前放一个随机"金丝雀值"，函数返回前检查是否被篡改 |
| **栈随机化（ASLR）** | 每次运行栈的位置不同，攻击者难以预测返回地址 |
| **NX bit** | 栈内存标记为不可执行，即使注入代码也无法运行 |

### 10.3 变长栈帧

`alloca()` 或变长数组（VLA）需要在编译期未知栈帧大小。编译器使用 `%rbp` 作为**帧指针**来追踪：

```
movq %rsp, %rbp       // 固定帧指针
subq %rdx, %rsp       // 动态分配变长空间
...
leave                  // movq %rbp, %rsp; pop %rbp（恢复栈帧）
```

## 十一、浮点代码（§3.12）

### 11.1 XMM 寄存器

x86-64 有 16 个 128 位 XMM 寄存器（`%xmm0`–`%xmm15`），用于浮点和 SIMD 运算。AVX 扩展到 256 位 `%ymm0`–`%ymm15`。

### 11.2 浮点传送

| 指令 | 功能 |
|:---:|------|
| `movss` | 传送单精度标量（4 字节）|
| `movsd` | 传送双精度标量（8 字节）|

### 11.3 浮点运算

| 指令 | 功能（双精度）|
|:---:|------|
| `addsd` | 加法 `D = D + S` |
| `subsd` | 减法 `D = D - S` |
| `mulsd` | 乘法 |
| `divsd` | 除法 |

浮点运算参数和返回值都通过 `%xmm0`–`%xmm7` 传递（与整数寄存器分开）。

### 11.4 浮点比较

| 指令 | 功能 |
|:---:|------|
| `ucomisd S, D` | 比较 `D` 和 `S`，设置条件码 ZF/CF |

之后用 `ja`/`jb`/`je` 等整数跳转指令读取结果。注意：若任一操作数为 NaN，ZF=CF=PF=1。

## 十二、重点总结 & 常见考题类型

### 12.1 核心原则一句话

> **C 代码的每种构造（变量、表达式、if/while/switch、函数调用、数组访问、结构体）都有一组固定的汇编指令模式。理解这些模式，就能"读懂"汇编代码。**

### 12.2 必会技能清单

1. **读寄存器**：给定汇编代码，说出每个寄存器的值变化
2. **操作数寻址**：`8(%rdi, %rsi, 4)` → 计算目标地址 = `%rdi + %rsi*4 + 8`
3. **条件码分析**：给定 `cmp`/`test` 结果，判断 `je`/`jg`/`jb` 是否跳转
4. **栈帧跟踪**：给定 `push`/`call`/`ret` 序列，画出 `%rsp` 和栈内容的变化
5. **C → 汇编翻译**：给一段 C 代码，写出对应的 x86-64 汇编
6. **汇编 → C 还原**：给一段汇编，还原为 C 代码
7. **定标寻址**：数组 `A[i][j]` 的寻址公式
8. **缓冲区溢出漏洞**：分析缓冲区大小，判断是否可能覆盖返回地址
