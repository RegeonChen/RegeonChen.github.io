## 一、概述

### 1.1 本章在全书中的位置

第 7 章回答一个核心问题：**`main.c` 里调了 `printf`，`printf` 在 `libc.so` 里——CPU 最后怎么找到它的？**

**链接（Linking）** 是将多个可重定位目标文件（`.o`）和库文件（`.a` / `.so`）合并成一个可执行文件的过程。它是编译流水线的最后一环，也是"程序映像"（program image）诞生的地方——链接决定了代码和数据在虚拟地址空间中的最终布局。

| 前驱章节 | 与第 7 章的关系 |
|:--------:|---------------|
| 第 2 章（← 依赖）| `.data` 段初始值的大小端存放；符号引用也是补码/十六进制值 |
| 第 3 章（← 依赖）| 重定位条目描述的是汇编指令中的地址占位符；PIC/GOT 是汇编层面的间接访问机制 |

| 后继章节 | 第 7 章为其提供了什么 |
|:--------:|-------------------|
| 第 8 章 ECF | `execve` 加载的程序映像是链接器生成的 ELF 可执行文件 |
| 第 9 章 虚拟内存 | 链接决定段布局 → 虚拟内存在 `execve` 时按 segment 映射；PIC 使共享库各进程共享同一物理页 |

## 二、编译器驱动程序与目标文件（§7.1–§7.3）

### 2.1 编译全过程回顾

CSAPP 第 1 章和第 7 章联合描述了 C 程序的完整生成过程。链接是最后一步：

```
main.c  ──cpp──→  main.i  ──cc1──→  main.s  ──as──→  main.o
sum.c   ──cpp──→  sum.i   ──cc1──→  sum.s   ──as──→  sum.o
                                                      │
                                             ┌─────"ld" 链接器─────┐
                                             │  合并所有 .o + 库    │
                                             └─────────┬───────────┘
                                                       ▼
                                                     prog (可执行文件)
```

`gcc` 实际上是一个**编译器驱动程序**，它依次调用：
- **cpp**（预处理，生成 `.i`）
- **cc1**（C 编译器，生成 `.s`）
- **as**（汇编器，生成 `.o`）
- **ld**（链接器，生成可执行文件 `prog`）

### 2.2 为什么需要链接？

没有链接，程序就需要把所有代码写在一个巨大的 `.c` 文件中。链接的好处：

1. **模块化**：每个 `.c` 独立编译，修改一个文件只需重编译该文件
2. **库复用**：`printf`/`malloc` 等只需写一次，被所有程序共享链接
3. **分离编译**：大型项目可以并行编译、增量构建

## 三、ELF 可重定位目标文件（§7.4–§7.5）

### 3.1 ELF 文件格式

ELF（Executable and Linkable Format）是 Linux/Unix 的标准目标文件格式。一个 ELF 可重定位目标文件 `main.o` 的结构如下：

```
┌────────────────────┐
│    ELF Header      │  ← 魔数、文件类型、机器类型
├────────────────────┤
│    .text           │  ← 已编译的机器代码
├────────────────────┤
│    .rodata         │  ← 只读数据（字符串常量、switch 跳转表等）
├────────────────────┤
│    .data           │  ← 已初始化、初值非零的全局/静态变量
├────────────────────┤
│    .bss            │  ← 未初始化或零初始化的全局/静态变量（不占磁盘空间）
├────────────────────┤
│    .symtab         │  ← 符号表：函数名、全局变量名及其属性
├────────────────────┤
│    .rel.text       │  ← .text 段的重定位信息
├────────────────────┤
│    .rel.data       │  ← .data 段的重定位信息
├────────────────────┤
│    .debug          │  ← 调试符号（gcc -g 时产生）
├────────────────────┤
│    .line           │  ← 源代码行号与机器指令的映射
├────────────────────┤
│    .strtab         │  ← 字符串表（各符号名称的字符串存储区）
├────────────────────┤
│ Section Header     │  ← 各段的元数据（偏移、大小等）
│     Table          │
└────────────────────┘
```

### 3.2 各小节功能速查

| 段名 | 内容 | 是否占 `.o` 磁盘空间 |
|:---:|------|:------------------:|
| `.text` | 函数编译后的机器码 | **是** |
| `.rodata` | `"hello"` 字符串常量、`switch` 跳转表、`const` 全局变量 | **是** |
| `.data` | `int global = 42;` 中的初始值 `0x2A 00 00 00` | **是** |
| `.bss` | `int uninit;` — 只记录大小（4 字节）| **否**（加载时由 OS 提供零页）|
| `.symtab` | 本模块**定义**和**引用**的所有符号 | **是** |
| `.rel.text` | `.text` 中需要被链接器修改的地址列表 | **是** |
| `.rel.data` | `.data` 中需要被链接器修改的地址列表 | **是** |
| `.strtab` | 所有符号名称的字符串池 | **是** |

## 四、符号与符号表（§7.5–§7.6）

### 4.1 什么是符号？

链接器眼中，模块 `main.o` 有三种符号：

| 类别 | 定义 | 示例 |
|:---:|------|:----|
| **全局符号（定义）** | 本模块定义、可被其他模块引用的函数或非静态全局变量 | `int global;`、`void func()` |
| **外部符号（引用）** | 本模块引用、由其他模块定义的函数或变量 | `extern int x;`、`printf` |
| **局部符号** | `static` 修饰的函数或变量——仅本模块可见 | `static int hidden;` |

> **关键区别**：C 语言的 `static` 在链接层面的含义是**隐藏**——将符号作用域限制在本模块内。两个不同的 `.o` 中可以各自有 `static int x;`，互不冲突。

### 4.2 符号表 `.symtab` —— 符号的"户口本"

`.symtab` 段是 ELF 文件中存放**所有符号信息**的表格，由汇编器 `as` 在汇编阶段生成。链接器阅读 `.symtab` 来知道每个模块"我提供了谁"和"我需要谁"。

#### 符号表条目的结构

```c
typedef struct {
    int      name;      // 符号名在 .strtab 中的偏移（不直接存字符串）
    char     type:4,    // 符号类型：函数 or 数据
             binding:4; // 绑定属性：LOCAL / GLOBAL / WEAK
    char     reserved;  // 保留
    short    section;   // 符号所在段的索引（.text=?   .data=?  UNDEF=0  ABS=特殊）
    long     value;     // 符号的偏移或地址
    long     size;      // 对象大小（字节）
} Elf64_Sym;
```

#### 各字段详解

| 字段 | 含义 | 关键取值 |
|:---:|------|---------|
| **binding** | 符号的作用域 | `LOCAL`（`static`，仅本模块可见）、`GLOBAL`（外部可见）、`WEAK`（弱符号）|
| **type** | 符号的种类 | `FUNC`（函数）、`OBJECT`（变量）、`NOTYPE`（未指定）|
| **section** | 符号定义在哪个段 | `1`=`.text`、`3`=`.data`、**`0`=`UNDEF`**（未定义——即外部引用）、`ABS`（绝对值，不应被重定位）|
| **value** | 符号的偏移 | 对于 `.o` 文件：距所在段起始位置的偏移；对于可执行文件：**绝对虚拟地址**（重定位后填入）|
| **size** | 符号占多少字节 | `sizeof(int)`=4、函数体长度等 |

#### 三类符号在 `.symtab` 中的体现

以 `main.o` 为例，假设它包含：

```c
int global = 42;              // 全局符号（定义）
static int hidden;             // 局部符号（static）
extern int printf(...);        // 外部符号（引用）
int main() { ... }            // 全局符号（定义）
```

它的 `.symtab` 大致如下（简化）：

| name | binding | type | section | value | 含义 |
|:----|:-------:|:----:|:------:|:-----:|------|
| `main` | GLOBAL | FUNC | `.text` | offset in `.text` | 我定义了一个全局函数 |
| `global` | GLOBAL | OBJECT | `.data` | offset in `.data` | 我定义了一个全局变量 |
| `hidden` | **LOCAL** | OBJECT | `.bss` | offset in `.bss` | 我定义了一个局部变量（static）|
| `printf` | GLOBAL | NOTYPE | **UNDEF** (0) | 0 | **我引用了别人定义的符号** |

> **关键观察**：`section = UNDEF` 的符号就是第 3 章所说的"外部引用"——它们不占本模块任何段的实际空间，但链接器必须在其他模块或库中找到它们的定义，否则报 `undefined reference`。

#### `.symtab` 不包含什么？

- **局部非静态变量**（函数内的 `int x;`）——这些在运行时存在于栈或寄存器中，不需要链接器操心
- **宏**——已被预处理器展开，编译器根本看不到
- **段名、文件名等元数据**——这些在其他表中（如 Section Header Table）

### 4.3 强符号与弱符号

| 类型 | 判定 | 示例 |
|:---:|------|:----|
| **强符号** | 已初始化的全局变量 + 函数定义 | `int x = 5;`、`void f(){}` |
| **弱符号** | 未初始化的全局变量（C99 后有些复杂，简单记为"无初始值"）| `int y;` |

**链接器处理强弱符号的三条规则**：

1. **不允许有多个同名的强符号** → 链接错误：duplicate symbol
2. **一个强符号 + 多个弱符号** → 选强符号（弱符号被覆盖）
3. **若全是弱符号** → 任选其一（大小以最大的为准）

```c
// lib1.c:  int x;          // 弱符号
// lib2.c:  int x = 5;      // 强符号
// → 链接后 x = 5，所有引用都解析到此强符号
```

### 4.4 常见的链接错误

| 错误类型 | 典型消息 | 成因 |
|:------:|---------|------|
| **重定义** | `multiple definition of 'x'` | 两个模块各定义了一个同名的强符号 |
| **未定义引用** | `undefined reference to 'foo'` | 所有模块都引用了 `foo`，但没有模块定义它 |
| **找不到库** | `cannot find -lxxx` | 链接时 `-L` 路径不对或未安装对应 `.a`/`.so` |

## 五、符号解析（§7.6）

### 5.1 链接器如何解析符号

链接器维护三个集合：

- **E**（Executable）：已确定定义、不会再有冲突的符号集合
- **U**（Unresolved）：已引用但尚未找到定义的符号集合
- **D**（Defined）：已扫描到的已定义符号集合（各模块的导出符号）

按命令行顺序扫描每个 `.o` 和 `.a`：

```
for each input file f:
    if f 是 .o:
        将 f 的符号定义加入 D
        若 f 的引用在 D 中有定义 → 解析
        否则 → 加入 U
    if f 是 .a (静态库):
        for each member m in f:
            if m 定义了 U 中的一个符号：
                链接 m，更新 D 和 U
```

### 5.2 **关键规则：链接顺序很重要**

静态库的链接遵循**单向按需提取**：只有被 U 中已有的未解析符号引用的 `.o` 才会从 `.a` 中被取出。因此：

```bash
# 错误：libcalc.a 中引用了 libmath.a 的符号，但 libmath.a 放在前面
gcc main.o -lmath -lcalc     # → undefined reference！

# 正确：库放在引用它的目标文件/库之后
gcc main.o -lcalc -lmath     # ✓ 或重复指定：gcc main.o -lcalc -lmath -lcalc
```

**经验法则**：库放在命令行末尾，被依赖的库放在依赖它的库**后面**；如有循环依赖则重复写。

## 六、重定位（§7.7）

重定位是链接的核心步骤，分两步。

### 6.1 步骤一：重定位段和符号定义

链接器将所有同类型段合并为一个聚合段，并为每个符号分配**运行时地址**：

```
main.o 的 .text ─┐
sum.o  的 .text ─┤──→ 可执行文件的 .text（合并后的整体）
libc   的 .text ─┘
                    每个函数和全局变量现在都有了确定的虚拟地址
```

### 6.2 步骤二：重定位段内的符号引用

修改 `.text` 和 `.data` 中引用符号的地址占位符——将占位符（全 0）替换为正确的运行地址。

链接器依赖**重定位条目（relocation entry）** 来知道改哪里、怎么改：

```c
typedef struct {
    long   offset;    // 引用在段中的偏移
    long   type:32,   // 重定位类型
           symbol:32; // 被引用的符号在 .symtab 中的索引
    long   addend;    // 常数加数
} Elf64_Rela;
```

### 6.3 两种重定位类型

#### R_X86_64_PC32 —— 相对寻址

用于 `call` 指令和 `%rip`-相对数据引用。计算公式：

```c
*refptr = (unsigned) (ADDR(r.symbol) + r.addend - refaddr);
```

其中 `refaddr = ADDR(s) + r.offset`（即引用的运行时地址）。

**为什么有个 -4 的 addend？**

```asm
e8 00 00 00 00           # call sum 的机器码
```

- `call` 指令本身 5 字节，`%rip` 在 `call` 执行时已指向下一条指令（即 `call` 起始地址 + 5）
- 但 `r.offset` 指向 `call` 指令中 `00 00 00 00` 占位符的第一个字节
- `refaddr = ADDR(.text) + r.offset` 指向占位符起始，而不是 `%rip` 在下一条指令时的值
- 因此需要加 `addend = -4` 来修正这个差异

#### 经典计算示例

```
已知：ADDR(main) = ADDR(.text) = 0x4004d0
     ADDR(sum)    = 0x4004e8
     r.offset = 0xf   (call 指令中占位符的偏移)
     r.symbol = sum
     r.type   = R_X86_64_PC32
     r.addend = -4

refaddr = ADDR(s) + r.offset = ADDR(.text) + 0xf = 0x4004d0 + 0xf = 0x4004df

*refptr = ADDR(sum) + (-4) - refaddr
        = 0x4004e8 - 4 - 0x4004df
        = 0x5

→ 最终机器码: e8 05 00 00 00  (call 的目标地址 = 下一条指令地址 + 5 = sum 的地址)
```

#### R_X86_64_64 —— 绝对寻址

用于数据段中对全局变量地址的引用（如 `int *p = &global;`）：

```c
*refptr = (unsigned) (ADDR(r.symbol) + r.addend);
```

不需减 `refaddr`，直接填入符号的绝对虚拟地址。

```
// int *bufp0 = &buf[0];
r.offset = 0x0
r.symbol = buf
r.type   = R_X86_64_64
r.addend = 0
ADDR(buf) = 0x601030

*refptr = 0x601030   → 对应地址处的 8 字节填为 30 10 60 00 00 00 00 00
```

### 6.4 重定位算法伪代码

```c
foreach section s {
    foreach relocation entry r in s's relocation section {
        refptr = s + r.offset;

        if (r.type == R_X86_64_PC32) {
            refaddr = ADDR(s) + r.offset;
            *refptr = (unsigned)(ADDR(r.symbol) + r.addend - refaddr);
        }
        if (r.type == R_X86_64_64 || r.type == R_X86_64_32) {
            *refptr = (unsigned)(ADDR(r.symbol) + r.addend);
        }
    }
}
```

## 七、可执行目标文件与加载（§7.8–§7.9）

### 7.1 可执行文件的 ELF 结构

可执行文件与 `.o` 文件结构相似，但多了 **Program Header Table（Segment 表）**：

```
可执行文件 prog:
┌────────────────────┐
│    ELF Header      │  ← e_entry = 入口点地址（_start）
├────────────────────┤
│ Program Header     │  ← execve 加载程序时用这个表（Segment 视角）
│     Table          │
├────────────────────┤
│    .text           │
│    .rodata         │  ← LOAD segment 0: R-X（只读 + 可执行）
├────────────────────┤
│    .data           │
│    .bss（不占空间）  │  ← LOAD segment 1: RW-（读写）
├────────────────────┤
│ Section Header     │
│     Table          │
└────────────────────┘
```

**关键字段**：ELF Header 中的 `e_entry` 指向 `_start`（由 `crt1.o` 提供），是程序第一条指令的虚拟地址。

### 7.2 Segment 表（Program Header Table）

可执行文件中，链接器把多个 section 合并为少数几个 **LOAD segment**，方便内核一次 mmap 就完成映射：

| Segment | 合并的 sections | 权限 | 后备存储 |
|:-------:|:--------------:|:---:|:-------:|
| LOAD 0 | `.text` + `.rodata` + `.init` | R-X | 可执行文件本身 |
| LOAD 1 | `.data` + `.bss` | RW- | `.data` 初始值来自文件；`.bss` 部分映射到零页 |

```
例：一个 prog 的 Program Header
LOAD off    0x0000000000000000
     vaddr  0x0000000000400000
     filesz 0x000000000000069c
     memsz  0x000000000000069c
     flags  r-x

LOAD off    0x0000000000000df8
     vaddr  0x0000000000600df8
     filesz 0x0000000000000228    ← .data 的实际大小（来自文件）
     memsz  0x0000000000000230    ← .data + .bss（.bss 在文件外，但需要映射）
     flags  rw-
```

**filesz < memsz 说明有 `.bss`**：文件只提供 `.data` 的初始值，`.bss` 部分在文件外被清零。

### 7.3 加载过程

当 `execve("./prog")` 被调用时（第 8 章 §8.5），加载器：

1. 读取 ELF Header 和 Program Header Table
2. 对每个 LOAD segment，用 `mmap` 映射到虚拟地址空间（第 9 章）——**不实际读入 DRAM**
3. `p_filesz` 部分从可执行文件映射；`p_memsz - p_filesz` 部分（`.bss`）映射到零页
4. 设置栈、`argc`、`argv[]`、`envp[]`
5. 跳转到入口点 `_start`

### 7.4 程序入口点与启动代码

程序的真正入口不是 `main`，而是 `_start`（由 `crt1.o` 定义）：

```
_start
  → __libc_start_main  (libc.so)
       → 初始化 C 运行时环境
       → 调用 main(argc, argv, envp)
       → 处理 main 的返回值
       → 调用 exit() 终止进程
```

## 八、动态链接共享库（§7.10–§7.12）

### 8.1 静态库 vs 动态库

| 特性 | 静态库 `.a` | 动态共享库 `.so` |
|:---:|:---------:|:-------------:|
| 链接时机 | 编译时复制到可执行文件 | 运行时由动态链接器加载 |
| 可执行文件大小 | **大**（拷贝了库代码）| **小**（只记录依赖）|
| 多个进程使用 | 各有一份独立副本 → 浪费 DRAM | **共享同一份物理页** |
| 库更新 | 需重新编译链接 | 更新 `.so` 即可（ABI 兼容时）|
| 加载速度 | 快（无需运行时解析）| 略慢（首次调用需解析）|

### 8.2 PIC —— 位置无关代码

动态库需要能被加载到任意虚拟地址。这就需要**位置无关代码（Position-Independent Code）**——代码中不能硬编码绝对地址。

PIC 用 **`%rip`-相对寻址**（第 3 章）实现：
- 代码永远只写"距离我当前指令多远"，而不是"去地址 0x7f..."
- 无论代码被加载到哪个虚拟地址，相对偏移始终不变

```asm
# PIC: 访问全局变量 global
movq global(%rip), %rax    # %rax = M[%rip + 相对偏移]
# 不管 .data 段被加载到哪个绝对地址，偏移量不变
```

### 8.3 GOT 与 PLT —— 延迟绑定的核心武器

动态库的函数地址要到**运行时**才知道——且不需要第一次调用前就解析全部函数。

#### GOT（Global Offset Table，全局偏移表）

GOT 是一个数组（在 `.data` 段），每个表项 8 字节，存储一个全局变量或函数的**运行地址**。

```
GOT[0]: 动态链接器的私有信息
GOT[1]: 动态链接器的 link_map 指针
GOT[2]: 动态链接器的解析函数地址
GOT[3]: printf 的实际地址（初始指向 PLT 桩的下一条指令）
GOT[4]: malloc 的实际地址
...
```

#### PLT（Procedure Linkage Table，过程链接表）

PLT 是一段"存根"代码（stub），在 `.text` 段中。每个动态库函数有一项 PLT 条目：

```asm
printf@PLT:
    jmpq   *GOT[3]          # 如果已解析 → 直接跳转 printf
    pushq  $index            # 压入 printf 在重定位表中的索引
    jmp    PLT[0]            # 跳转到动态链接器的解析入口

PLT[0]:                      # 动态链接器的公共入口
    pushq  GOT[1]            # 压入 link_map
    jmpq   *GOT[2]           # 跳转到 _dl_runtime_resolve
```

#### 延迟绑定（Lazy Binding）的完整流程

```
第一次调用 printf:
    call printf@PLT
      → jmpq *GOT[3]       # GOT[3] 初始指向 PLT 的下一条指令（pushq $index）
      → pushq $index        # 告诉解析器：请解析第 $index 号函数
      → jmp PLT[0]          # 跳转到动态链接器
      → 动态链接器查找 printf 在 libc.so 中的真实地址
      → 将真实地址写入 GOT[3]
      → 调用 printf

第二次及以后调用 printf:
    call printf@PLT
      → jmpq *GOT[3]       # GOT[3] 已是 printf 的真实地址 → 直接跳转
      # 不再触发 PLT 桩！
```

**对比**：

| | 首次调用 | 后续调用 |
|:--|:------|:------|
| 开销 | 需要动态链接器介入解析 | 仅一次间接跳转（~2-3 个周期）|
| 思想 | 懒解析——不调不解 | 已解析的直接命中 |

### 8.4 运行时的动态链接

```bash
ldd prog    # 查看 prog 依赖哪些 .so
```

在运行时，动态链接器 `ld-linux.so`：
1. 将依赖的 `.so` 映射到虚拟地址空间
2. 执行所有 `.so` 的初始化代码
3. 修正 GOT 中已解析数据的表项
4. 将控制权转交 `main`
5. PLT 的延迟绑定：函数的 GOT 表项初始指向 PLT 桩 → 首次调用时解析并重写 GOT

#### GOT/PLT 经典例题

**例题 1（首次 vs 后续调用）**：程序首次调用 `malloc` 和第二次调用 `malloc`，GOT 表项分别指向哪里？

```
首次调用 malloc:
  call malloc@PLT
    → jmp *GOT[malloc]        # GOT[malloc] 初始指向 PLT 桩的下一条 (pushq $index)
    → pushq $index
    → jmp PLT[0]              # 触发动态链接器
    → 动态链接器：找到 malloc 在 libc.so 中的真实地址 0x7f... → 写入 GOT[malloc]
    → 跳转执行 malloc

第二次调用 malloc:
  call malloc@PLT
    → jmp *GOT[malloc]        # GOT[malloc] 现在 = 0x7f...（malloc 真实地址）
    → 直接跳转！不再走 PLT 桩
```

**例题 2（GOT 更新前后对比）**：第一次调用前，`objdump` 看到 `GOT[printf]` 的值是什么？调用后变成什么？

> 调用前：`GOT[printf] = PLT 桩中 pushq $index 指令的地址`（即 `printf@PLT` 的第 2 条指令地址），动态链接器靠这个"回跳"来知道是哪个函数需要解析。
>
> 调用后：`GOT[printf] = libc.so 中 printf 的真实虚拟地址`（如 `0x7f...`），以后直接跳转。

**例题 3（识别 GOT/PLT 的段归属）**：为什么 PLT 在 `.text` 段而 GOT 在 `.data` 段（或 `got.plt` 段）？

> PLT 是**可执行代码**（`jmp`、`push`、`jmp` 指令）→ 必须放在只读+可执行的 `.text` 段，所有进程共享同一物理页。
>
> GOT 是**数据表**（存储运行时的解析地址）→ 必须放在可读写的 `.data` 段，**每个进程私有一份**（因为不同进程的 `.so` 可能被映射到不同虚拟地址，GOT 中存的是各自映射后的绝对地址）。初始时所有进程的 GOT 内容相同（都指向 PLT 桩），但首次调用后各自填充不同的值。这种设计恰好利用了第 9 章的 COW 机制——GOT 页面初始共享，修改时触发 COW 创建私有副本。

## 九、库打桩 Library Interpositioning（§7.13）

库打桩允许在**不修改源代码**的情况下拦截和替换库函数调用。分三个层次：

| 层级 | 实现方式 | 何时生效 |
|:---:|---------|:------:|
| **编译时** | `#define` 宏 + 头文件 | 编译阶段拦截 |
| **链接时** | `--wrap` 标志（ld 选项）| 链接阶段替换 |
| **运行时** | `LD_PRELOAD` 环境变量 | 动态链接器先加载指定 `.so` |

```bash
# 运行时打桩：用自己的 malloc 替换 libc 的 malloc
gcc -shared -fpic mymalloc.c -o mymalloc.so
LD_PRELOAD="./mymalloc.so" ./prog
```

应用场景：内存泄漏检测、性能 profiling、安全沙箱、代码注入。

## 十、重点总结 & 常见考题类型

### 10.1 核心原理一句话

> **链接器将多个 `.o` 和 `.a`/`.so` 合并为可执行文件：先解析符号（强弱符号规则），再重定位（PC32 相对 / 绝对），最终生成的程序映像由 `execve` 映射到虚拟地址空间。动态库通过 PIC + GOT + PLT + 延迟绑定实现位置无关和多进程共享。**

### 10.2 必会技能清单

1. **识别符号类型**：给定变量/函数声明，判断是强/弱符号、全局/局部、定义/引用
2. **强弱符号规则**：给定多个 `.c` 中的同名符号，判断链接结果或报错
3. **链接顺序分析**：判断给定命令行会不会出现 undefined reference
4. **ELF 段识别**：说出 `.text`/`.data`/`.bss`/`.symtab`/`.rel.text`/`.rel.data` 各存什么
5. **重定位计算**：
   - PC32 相对寻址：`*refptr = ADDR(sym) + addend - refaddr`
   - 绝对寻址：`*refptr = ADDR(sym) + addend`
6. **GOT/PLT 流程**：解释 printf 首次调用→后续调用的完整过程
7. **PIC 原理**：为什么 `.so` 可以加载到任意虚拟地址
8. **静态库 vs 动态库**：trade-off（大小、内存共享、更新、启动速度）
9. **e_entry 指向 _start**：不是 main！理解启动代码做了什么

### 10.3 与第 3 章 + 第 9 章的桥梁

```
第3章 汇编             第7章 链接              第9章 虚拟内存
  call sum ↓              .rel.text 条目 ↓          execve ↓
  e8 00 00 00 00   →    R_X86_64_PC32    →    将 LOAD segment
  (占位符)              填入真实偏移 0x05             mmap 到虚拟地址空间
                        (PC32 重定位)               (按需调页)

  movq $0, x(%rip) →    重定位 x 的地址     →    x 的页表映射建立
```

### 10.4 常见混淆点澄清

| 易混淆概念 | 正确理解 |
|:---------|---------|
| `.bss` 不在文件中 | `.bss` 在可执行文件/`.o` 中只记录大小，不占磁盘空间。其零值由加载器通过映射零页提供 |
| `static` 的双重含义 | 在函数内 = 静态局部变量；在函数/变量声明前 = 符号作用域限制在本模块（链接层面隐藏） |
| `_start` ≠ `main` | `_start` 是 ELF 的入口点，来自 `crt1.o`，负责初始化运行时再调用 `main` |
| 共享库的代码共享 vs 数据独立 | 代码段 `.text` 所有进程共享同一物理页；数据段 `.data`/`.bss` 通过 COW 每进程各自独立 |
| PLT 在 `.text`，GOT 在 `.data` | PLT 是只读代码段内的桩代码，GOT 是数据段内的地址表——这是故意分开的，保证代码段只读共享 |

### 10.5 静态链接 + 动态链接的经典混合考法

#### 考法 1：链接时选择 vs 运行时加载

```
gcc main.c -L. -lmylib -o prog     # libmylib.a（静态库）→ 代码被复制进 prog
                                    # libmylib.so 也存在于 . → 链接器优先选静态版本
```

若 `libmylib.a` 和 `libmylib.so` 同时存在，`-lmylib` 默认链接 `.so`。要强制静态链接用 `-static -lmylib`。**考点**：为什么同一个符号可能被解析到不同地址？答案——静态链接时 `mylib_func` 的地址已被链接器填入 `prog` 的 `.text`（可直接跳转，无需 PLT/GOT）；动态链接时对 `mylib_func` 的调用走 `call mylib_func@PLT`（通过 GOT 间接，首次触发延迟绑定）。

#### 考法 2：混合模块的符号解析冲突

```
假设：
  main.o 中引用外部符号 x（未定义）
  libstatic.a 定义了强符号 x = 5
  libdynamic.so 也定义了强符号 x = 10（FULL RELRO 下 GOT 直接填充）
```

**问题**：最终 `main` 中 `printf("%d", x)` 输出 5 还是 10？

> 输出 5。链接器处理符号解析时，静态库的符号和动态库的符号在链接阶段就已经被决策：静态库中的 `.o` 被提取并合并进可执行文件，其强符号 `x` 填入 `.data` 段。动态库此时只留下符号引用记录，运行时不覆盖已解析的静态符号。

#### 考法 3：PIC vs 非 PIC 的可执行文件

```
# 两个 prog，一个 PIC，一个非 PIC：
gcc -fno-pic main.c -o prog_nopic     # 非 PIC：代码中地址写死
gcc -fpic main.c -o prog_pic          # PIC：代码用 %rip-相对
```

**问题**：`objdump -d` 两个 `prog` 的 `main` 函数，调用 `printf` 的代码有什么不同？

> 两个都用 `call printf@PLT`（因为 `printf` 来自动态库 `libc.so`，无论如何都要走 PLT）。**区别在于** `prog_nopic` 中引用自己模块的全局变量 `int x` 是绝对地址 `movl $0x601040, %eax`（链接时已确定，硬编码）；`prog_pic` 中是 `movl x(%rip), %eax`（`%rip`-相对，可在任意地址运行）。**考点**：PIC 的可执行文件可以被 ASLR 随机放置到任意虚拟地址，而非 PIC 的只能放在链接器指定的固定地址。

#### 考法 4：LD_PRELOAD 打桩 —— 静态调用者 vs 动态被调用者

```c
// prog.c (动态链接 libc):
#include <stdlib.h>
int main() { void *p = malloc(16); free(p); }

// mymalloc.c (编译为 mymalloc.so):
void *malloc(size_t s) { /* 追踪 + 调用真实 malloc */ }
```

```bash
gcc -shared -fpic mymalloc.c -o mymalloc.so
LD_PRELOAD=./mymalloc.so ./prog    # main 中的 malloc 被拦截！
```

**问题**：如果把 `prog` 改为静态链接 `libc.a`（`gcc -static prog.c`），`LD_PRELOAD` 还能拦截 `malloc` 吗？

> **不能**。静态链接时 `malloc` 的代码已经被**复制进 `prog` 的可执行文件**——它是 `prog` 自己的 `.text` 中的一块，不再是"外部符号"。运行时没有 GOT/PLT，没有间接跳转，`main` 中 `call malloc` 直接跳转到 `prog` 内部的 `malloc` 地址。`LD_PRELOAD` 只能影响动态符号的解析（即 GOT/PLT 路径），对静态链接的函数完全无效。

#### 考法 5：静态库和动态库对 GOT 的影响

| 场景 | GOT/PLT 存在吗？ |
|:-----|:----------------:|
| 所有库都是静态 `.a` 链接 | **否** —— 生成的 `prog` 不依赖任何 `.so`，没有 GOT/PLT 表，也没有动态符号表 |
| 部分或全部使用 `.so` | **是** —— `prog` 有 GOT/PLT 表，但**只有那些最终来自 `.so` 且未被 `.a` 提前解析的符号**才拥有对应表项。被 `.a` 提前解析的符号沦为普通本地函数，不占用 GOT |

**问题（综合）**：`gcc main.o -lstatic -lshared -o prog`，其中 `lshared.so` 定义了函数 `f`，但 `libstatic.a` 中也有一份 `f`。`main` 调用 `f()` 时走 PLT 还是直接跳转？`objdump -R prog` 能看到 `f` 的重定位条目吗？

> 走**直接跳转**（无 PLT）。链接器先扫描静态库 `libstatic.a`，发现它定义了 `f` 并且 `f` 正在 U（未解析符号集合）中，于是将 `libstatic.a` 的实现提取并合并进 `prog`。此后 `f` 不再在 U 中，即使 `libshared.so` 也定义了 `f`，`f` 已经被静态解析，`objdump -R` 中不会出现 `f` 的 GOT 重定位条目。这正是 §5.2 的"静态库按需提取 + 动态库只填充尚未解析的符号"的组合效果。
