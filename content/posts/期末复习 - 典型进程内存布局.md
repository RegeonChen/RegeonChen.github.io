## 〇、前置知识：

#### 1. 可执行文件存储在哪里？

> **可执行文件**（例如你编译生成的 `prog`）是一个**普通的磁盘文件**，通常存放在持久化的文件系统中（比如 Linux 的 ext4、XFS，或者 Windows 的 NTFS）。它位于**硬盘（HDD）或固态硬盘（SSD）** 上，即使断电内容也不会丢失。

- 当你运行 `./prog` 时，操作系统（通过加载器）**不会**把整个文件读入物理内存，而是将其中的代码段（`.text`）、只读数据段（`.rodata`）、已初始化数据段（`.data`）等映射到进程的虚拟地址空间中
- 较为特殊的是 `.bss` 段，在可执行文件中**不占用实际空间**（只记录了大小），它的初始零值由内核在加载时通过虚拟内存机制提供（通过映射到一个全局零页）。

## 一、虚拟内存、物理内存、Swap Space、持久化文件

#### 1. 虚拟内存的概念

一个关于虚拟内存的较为正确的理解：

>虚拟内存是一个抽象的地址空间，其中每个“有效”页面都**关联到一个后备存储**（要么是某个文件，要么是 swap space）。物理内存则是这些页面的**缓存**。
>
>**MMU（Memory Management Unit）**：CPU中负责“虚拟地址→物理地址”转换的硬件组件。

#### 2. 虚拟内存的大小

- 虚拟内存的“大小”通常指**虚拟地址空间的范围**，由处理器的地址线宽度决定（例如 64 位系统理论上有 $2^{64}$ 字节地址空间）。但进程实际能使用的虚拟地址空间远小于这个值，由操作系统分配的区域（代码、数据、堆、栈等）组成。
- 更相关的概念是**已分配的虚拟内存页面总量**（committed memory）。这些页面有些对应文件（可执行文件、共享库、内存映射文件），有些是匿名内存（`.bss`、堆、栈、COW 产生的私有页）。

#### 3. Swap Space

磁盘上的一块专用区域（可以是分区或文件），作为**虚拟内存的后备存储**之一，用于存放**匿名页**的内容。

##### 核心作用

- **扩展“内存”容量**：当物理内存不足时，操作系统将一部分不常访问的匿名页（如堆、栈、COW 产生的私有页）换出到 swap space，释放物理内存给更急需的页面。
- **支持休眠/挂起**：将整个内存镜像写入 swap space，实现“休眠到磁盘”（hibernate）

##### 关键特点

- **临时性**：系统重启后 swap 中的内容失效。
- **性能**：磁盘 I/O 速度远慢于物理内存，过度使用 swap 会导致系统性能严重下降。

#### 4. 持久化文件

与 **swap space（交换空间）** 相对的概念，用于存放 **虚拟内存中“文件后备页”（file-backed pages）的原始数据**。具体来说，它主要存储以下内容：

| 数据类别                              | 对应的段 / 内容                                          | 存储在哪个持久化文件？         |
| :------------------------------------ | :------------------------------------------------------- | :----------------------------- |
| **可执行程序的机器指令**              | `.text` 段（代码段）                                     | 可执行文件本身（如 `prog`）    |
| **只读数据（常量）**                  | `.rodata` 段（字符串常量、`const` 全局变量等）           | 可执行文件                     |
| **已初始化且初值非零的全局/静态变量** | `.data` 段的**初始内容**（例如 `int x = 42;` 中的 `42`） | 可执行文件                     |
| **动态链接共享库的代码与只读数据**    | 共享库的 `.text` 和 `.rodata`                            | 共享库文件（如 `libc.so`）     |
| **动态链接共享库的已初始化数据**      | 共享库的 `.data` 段初始内容                              | 共享库文件                     |
| **通过 `mmap` 映射的普通文件内容**    | 映射的文件数据（例如一个配置文件、数据库文件）           | 被映射的普通文件（由用户指定） |

**关键点**：持久化文件**不存储**以下内容：

- `.bss` 段（未初始化或零初始化的全局变量）——这些在可执行文件中只记录大小，不占用实际空间。
- 堆、栈上的数据（这些是运行时产生的匿名内存）。
- 修改过的 `.data` 或 `.bss` 页（COW 后产生的私有脏页）——这些已脱离原始文件，转而依赖 swap space。

```

+----------------------------------+
|          内核空间（用户不可读）      |
+----------------------------------+
|              栈 (stack)           |  ← 运行时向下增长，存放局部变量、返回地址
+----------------------------------+
|              共享库映射区域         |
+----------------------------------+

+----------------------------------+

+----------------------------------+
|      .data 段（已初始化静态数据）    |  ← 固定初始值
+----------------------------------+
|      .rodata 段（只读数据）         |  ← 字符串常量、只读全局变量
+----------------------------------+
|      .text 段（代码/指令）          |  ← 只读、可执行
+----------------------------------+
|              保留/低地址区域         |
低地址
```

#### 各段的作用与生命周期

| 段名      | 存储内容                                        | 生命周期             | 在目标文件/可执行文件中是否占磁盘空间              |
| --------- | ----------------------------------------------- | -------------------- | -------------------------------------------------- |
| `.text`   | 机器指令（函数代码）                            | 整个程序运行期间     | 是（从源文件生成）                                 |
| `.rodata` | 只读数据：字符串常量、`const` 全局变量          | 整个程序运行期间     | 是                                                 |
| `.data`   | 已初始化且初始值非零的全局/静态变量             | 整个程序运行期间     | 是（初始值存储在文件中）                           |
| `.bss`    | 未初始化的全局/静态变量 或 初始化为零的全局变量 | 整个程序运行期间     | **否**（只记录大小和位置，加载时由系统分配并清零） |
| 堆        | 动态分配的内存（`malloc` 等）                   | 从分配开始到显式释放 | 运行时产生，不在文件中                             |
| 栈        | 局部变量、函数调用帧、返回地址等                | 函数调用期间         | 运行时产生，不在文件中                             |

## 三、各段与持久化文件 / swap space 的对应关系

在 Linux / Unix 系统中，每一个已分配的虚拟页面，都有一个**后备存储**——即当物理内存不足需要换出该页时，系统将它写回的地方：

- **文件后备页（File-backed page）**：页的内容最初来自磁盘上的某个文件（可执行文件、共享库、普通数据文件等）。当内存紧张需要换出时，**干净的（未修改的）页可以直接丢弃**，下次访问时再从文件读回；修改过后的**脏页**（如 COW 产生的 `.data` 副本），不再与原始文件同步，系统必须把它当作匿名页处理，通常不能写回只读的可执行文件，因此需要 **swap space** 来保存这些修改。

- **匿名页（Anonymous page）**：没有对应的磁盘文件（例如 `.bss`、堆、栈、COW 复制出来的页）。这类页**只能换出到 swap space**，且换入时也从 swap space 读回。

具体到各段：

| 段名称              | 持久化文件？                                         | 使用 swap space？                                            |
| ------------------- | ---------------------------------------------------- | ------------------------------------------------------------ |
| `.text`             | **是**（来自可执行文件的代码段，只读）               | 基本**不需要** swap：干净的页直接丢弃，下次从文件重读；没有脏页。 |
| `.rodata`           | **是**（来自可执行文件的只读数据段）                 | 不需要 swap（同上）。                                        |
| `.data`（已初始化） | **是**（来自可执行文件的数据段，初始内容从文件复制） | **可能需要**：如果进程修改了 `.data` 中的变量（脏页），且该页是私有的（非 `MAP_SHARED`），则修改后的页不再与原始文件同步，变成“匿名页”，换出时需写入 swap space。 |
| `.bss`              | **否**（可执行文件只记录大小，没有实际内容）         | **是**：初始为全零，之后可能被修改。所有 `.bss` 页都是匿名页，换出时必须写入 swap。 |
| 堆（heap）          | **否**（动态分配）                                   | **是**：匿名页，依赖 swap。                                  |
| 栈（stack）         | **否**（运行时创建）                                 | **是**：匿名页，依赖 swap。                                  |

> **重要说明**：对于 `.data` 段，如果程序从不修改全局变量，那么这些页始终是“干净的文件后备页”，可以丢弃并从可执行文件重读，**不占用 swap**。一旦修改，那页就变成了“私有脏页”，后续换出需要 swap。这个转换也正是 COW 机制发挥作用的地方。

## 四、各段修改前后的内存行为

- **磁盘（包括可执行文件和 swap space）** 是虚拟内存的“低速大容量主存”。
- **物理内存（DRAM）** 是虚拟内存的“高速缓存”。
- 内存管理单元（MMU）和操作系统共同维护**页表**，记录每个虚拟页是否在物理内存中（缓存命中），如果不在则触发缺页异常，从磁盘（文件或 swap）换入。

| 页面类型                      | 后备存储（磁盘位置）                                         | 内存行为                                                     |
| ----------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| `.text` / `.rodata`（未修改） | 可执行文件                                                   | 干净页面，可以随时丢弃，再次访问时从可执行文件重读（缓存从磁盘填充） |
| `.data` 初始内容（未修改）    | 可执行文件                                                   | 同上                                                         |
| `.data` 修改后（COW 私有页）  | 变成匿名页 → 后备存储变为 **swap space**                     | 脏页，换出时必须写入 swap，换入时从 swap 读回                |
| `.bss`（未修改）              | 无实际磁盘存储（由零页提供零值，零页本身是内存中特殊的只读页） | 从未换出（因为零页全局共享，不会变脏）                       |
| `.bss` 修改后（COW 私有页）   | 匿名页 → 后备存储为 **swap space**                           | 与修改后的 `.data` 类似，换出/换入 swap                      |
| 堆、栈（匿名页）              | **swap space**                                               | 分配时为零页或未初始化；脏页换出到 swap                      |

**注**：可执行文件本身是只读的代码和只读/初始数据的来源。修改过的全局变量（`.data` 或 `.bss`）会脱离可执行文件，转而依赖 swap space。

## 五、COW 写时复制机制

**写时复制（Copy-on-Write, COW）** 机制，和我们之前讨论的 `.data`、`.bss` 段以及进程内存布局有非常紧密的联系，尤其是在 **`fork` 创建子进程** 以及 **共享库的全局变量处理** 中。

COW 本身是**缺页异常处理**和**物理页管理**的行为，**不直接触发磁盘 I/O**。但是，COW 的结果会影响后续的页面换入/换出（磁盘 I/O）。

#### 1. COW 的基本思想

- **写时复制** 是一种优化策略：**多个进程可以共享同一块物理内存，直到其中某个进程试图写入该内存时，系统才为它复制一个私有的副本**。
- 优点：大大减少了不必要的内存复制，节省物理内存，加快进程创建速度。

在 CSAPP 中，COW 最典型的应用场景是：

1. **`fork()` 创建子进程**
2. **共享库的全局变量**（例如通过动态链接加载的 `.so`）

---

#### 2. COW 与 `.bss` 段（未初始化/零初始化的全局变量）

你可能会问：`.bss` 段的内容全是零，那多个进程的 `.bss` 可以共享吗？  

**答案是：可以，而且系统正是这样做的，通过 COW 实现。**

##### 经典流程（以 `fork()` 为例）：

1. 父进程运行  
   - 它的 `.bss` 段（比如 `bufp1`）在物理内存中占用一个 **只读的零页**（内核维护一个全局的「零页」）。
   - 初始时，父进程对 `.bss` 页的访问不会触发复制，因为物理内存中本来就是一个全零的只读页面。
2. 父进程调用 `fork()`  
   - 子进程被创建，**不复制父进程的任何物理内存页**。
   - 子进程的页表项**被设置为指向父进程的同一物理页**，并且所有页（包括 `.text`、`.data`、`.bss`、堆、栈）都被标记为 **只读**。
   - 操作系统记录这些页为 “写时复制” 页。
3. **任一进程首次写入 `.bss` 变量**（以子进程首次写入为例）
   - CPU 检测到页表项中的“只读”标志，触发 **缺页异常（page fault）**。
   - 内核处理异常时发现该页属于 COW 区域：  
     - 在物理内存中**分配一个新的页**（大小通常为 4KB）。  
     - **将旧页的内容**（全零或父进程修改过的内容）**复制到新页**。  
     - 修改触发异常的进程的页表项，指向新页，并设为 **可读写**。此时：子进程的这个 `.bss` 页已经不是零页，而是一个**私有的、可写的匿名页**。这个新页的内容 **不在任何持久化文件中**，未来如果内存紧张需要换出，必须写入 **swap space**。
   - 进程重新执行写入指令，这次成功。
4. **另一个进程继续使用自己的副本**，互不干扰。
5. 内存压力导致换出（swap out）
   - 如果系统内存不足，内核会选择一些页换出到磁盘。
   - 对于子进程的这个私有 `.bss` 页（脏匿名页）：
     - 内核将其内容写入 **swap 分区或 swap 文件**。
     - 在页表中标记为“未驻留”，记录 swap 位置。
     - 释放物理页。
   - **发生了磁盘 I/O（写入 swap）**。
6. 后续再次访问该页（换入，swap in）
   - 子进程再次访问该数据，触发缺页。
   - 内核从 swap 空间读回内容到新物理页。
   - **发生了磁盘 I/O（读取 swap）**。
   - 注：父进程如果从未修改 `.data`，它的 `.data` 页始终是文件后备，可以丢弃并从可执行文件重读（不需要 swap）。

结论：`.bss` 段在物理内存中**最初被所有进程共享**（共享一个零页），**写时复制** 保证了每个进程在修改自己的 `.bss` 变量时获得私有副本。

这与 `.data` 段（已初始化且非零）的处理类似：  

- 父进程的 `.data` 段有具体的初始值。  
- `fork` 后，父子进程共享这些 `.data` 页（标记只读）。  
- 一旦某个进程修改 `bufp0`，也会触发 COW，得到一个私有副本。

实际上，除了 `text` 和 `.rodata` 之外，`.data`、`.bss`、堆和栈在 `fork()` 创建子进程时，均用到了COW机制。

#### 3. COW 的其他重要场景

> 该部分详见其他文件

1. **共享库（动态链接）的全局变量**

   当一个共享库（如 `libc.so`）被多个进程加载时：

   - 共享库的 **代码段（`.text`）** 是纯只读的，永久共享，无需 COW。
   - 共享库的 **数据段（`.data` 和 `.bss`）** 则是 **“每个进程一份副本”**，但系统也采用 COW 来延迟复制：  
     - 初始时，所有进程的页表项指向同一个物理数据页（只读）。  
     - 当某个进程尝试修改库的全局变量时，触发 COW，为该进程创建私有副本。  
     - 这样保证了库的全局变量在语义上是 **进程私有** 的，同时节省物理内存。

   这正是 **动态链接器** 利用 COW 实现“多个进程共享同一个动态库的代码，但各自拥有独立的数据”的经典方法。

2. **内存映射文件（`mmap`）的私有映射（`MAP_PRIVATE`）**
   - 用 `mmap` 以 `MAP_PRIVATE` 方式映射文件时，进程可以读取文件内容，修改只在内存中进行，不会写回文件。  
   - 这个修改同样是通过 COW 实现的：多个进程映射同一个文件时，共享物理页，一旦某个进程写入，才复制。
