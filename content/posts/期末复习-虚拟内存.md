## 二、虚拟内存——核心概念（§9.1–§9.2）

### 2.1 虚拟内存是什么

> 虚拟内存是一个抽象的地址空间，其中每个"有效"页面都**关联到一个后备存储**（要么是某个文件，要么是 swap space）。物理内存则是这些页面的**缓存**。

```
CPU 发出虚拟地址 (VA)
       ↓
    ┌─────┐
    │ MMU │  Memory Management Unit（CPU 内部硬件组件）
    └──┬──┘
       │  查 TLB → 查页表
       ↓
   物理地址 (PA)
       ↓
   主存 DRAM（物理内存是虚拟内存的"缓存"）
```

**MMU（Memory Management Unit）**：CPU 中负责"虚拟地址 → 物理地址"转换的硬件组件。每次访存都要经过它。

### 2.2 关键概念区分

| 概念 | 含义 | 类比 |
|:---:|------|------|
| **虚拟地址空间** | 每个进程看到的全部地址范围，由地址线宽度决定（64 位理论有 $2^{64}$ 字节）| 每个人都以为自己独占一栋楼 |
| **虚拟内存** | 已分配的虚拟页面的集合，每个有效页面有后备存储 | 实际使用的房间 |
| **物理内存（DRAM）** | 虚拟页面的缓存——只缓存活跃使用的页面 | 楼里实际住了人的房间 |
| **后备存储** | 页面不在 DRAM 时的老家（文件或 swap）| 东西没在房间，就在仓库里 |

### 2.3 虚拟内存的大小

- 虚拟内存的"大小"通常指**虚拟地址空间的范围**，由处理器地址线宽度决定。但进程实际使用的远小于此值。
- 更相关的概念是**已分配的虚拟内存页面总量**（committed memory）：有些对应文件（代码、共享库），有些是匿名内存（`.bss`、堆、栈、COW 私有页）。

### 2.4 虚拟内存的三大用途

| 用途 | 机制 | 效果 |
|:---:|------|------|
| **作为缓存的工具** | 主存是磁盘的缓存，缺页时调入 | "看上去"比实际 DRAM 大得多 |
| **作为内存管理的工具** | 每进程独立页表，`mmap` 映射文件 | 进程隔离 + 简化链接和加载 |
| **作为内存保护的工具** | PTE 中的权限位（R/W/U）+ 内核空间隔离 | 防止越界访问、防止篡改内核 |

## 三、虚拟内存作为缓存的工具（§9.3）

### 3.1 主存是磁盘的"缓存"——与第 6 章的直接对应

| 第 6 章概念 | 第 9 章对应 | 区别 |
|:---------:|:---------:|------|
| 缓存块 64B | **页面 4KB** | 页面是缓存的"块" |
| L1/L2/L3 (SRAM) | **主存 DRAM** | DRAM 是磁盘的"高速缓存" |
| 主存 DRAM | **磁盘（SSD/HDD）** | 磁盘是后备存储 |
| Tag / Set Index / Offset | **VPN = Virtual Page Number / VPO** | 地址翻译的字段名不同 |
| Valid bit | **PTE 有效位** | 概念完全相同 |
| 命中 / Miss | **页命中 / 缺页（Page Fault）** | 不命中叫"缺页" |
| LRU 替换 | **页面置换算法（LRU/Clock）** | OS 软件管理，硬件只管报告 |
| 写回 | **脏页写回** | 完全相同的理念 |
| 全相联（S=1）| **虚拟内存是全相联** | 避免冲突不命中——缺页代价太大 |

### 3.2 页面的状态

数据在"主存 ↔ 磁盘"的缓存体系中有四种可能状态：

```
分配了虚拟页？
   ├── 未分配（unallocated） → 不在虚拟内存中，不可访问
   └── 已分配（allocated）
         ├── 缓存在 DRAM 中 → 页命中（hit）
         └── 未缓存在 DRAM 中
               ├── 干净页（未修改过的文件后备页） → 可直接丢弃，从文件重读
               └── 脏页（匿名页 / 修改过的 .data） → 必须写回磁盘（swap）
```

| 页面状态 | 在 DRAM？ | 换出时行为 | 典型例子 |
|:-------:|:--------:|----------|---------|
| 未分配 | — | — | 未 `malloc` 的堆区域 |
| 已分配 · 缓存在 DRAM | ✓ | — | 正在活跃使用的栈、代码 |
| 干净的文件后备页 | ✗ | 直接丢弃，从文件重读 | 未修改的 `.text`、`.rodata`、`.data` |
| 脏页（匿名/修改后的 .data）| ✗ | **必须写回 swap space** | `.bss`、堆、栈、COW 私有页 |

### 3.3 按需调页（Demand Paging）—— "不访问就不加载"

虚拟内存的缓存模型与普通 CPU 缓存有一个根本区别：**虚拟内存是"按需"的——页面只在被访问时才实际加载到物理内存**。

```
execve("./prog") 加载程序时：
  ├── 建立页表映射：虚拟地址 → (待定)物理页
  ├── PTE 的 Valid 位 = 0（标记为"不在物理内存中"）
  └── 仅设置好虚拟地址空间布局，不实际拷贝任何代码或数据到 DRAM

首次执行 .text 中的指令时：
  ├── CPU 发出 VA → 查 TLB → miss → 查页表 → PTE Valid=0
  ├── 缺页故障！（第 8 章 §8.1 故障异常）
  ├── OS 缺页处理程序：从 ELF 文件读入 4KB 页面到 DRAM
  ├── 更新 PTE：Valid=1，填入 PPN
  └── 返回重试指令 → 这次命中，指令正常执行

首次访问全局变量时：
  ├── .data：从 ELF 文件读入初始值所在的页面
  ├── .bss：映射到全局零页（全零物理页），写入时 COW 分离
  └── 栈：首次 push/call 触发缺页 → OS 分配匿名零页
```

**为什么不用"全加载"（eager loading）？**

| 策略 | 做法 | 问题 |
|:---:|------|------|
| **全加载** | `execve` 时把整个可执行文件 + 所有依赖库一次性读入 DRAM | 大量页面加载后根本不会被访问（如错误处理代码、冷门功能），白白浪费 DRAM 和 I/O 带宽 |
| **按需调页** | 只在实际第一次访问时加载那一页 | 启动极快，内存利用高效——这正是现代 OS 选择按需调页的原因 |

**按需调页使得：**
- 程序启动飞快（`execve` 几乎不做 I/O，只是建立页表）
- 一个大程序只用到的部分才占用 DRAM（不用的代码段永远不会被加载）
- COW 的"偷懒拷贝"也是基于同一机制——`fork()` 不真的复制内存，只标记只读，写入时才按需复制

> 按需调页 + 缺页故障 + COW = 虚拟内存"既大又快又隔离"的三个引擎。按需调页是虚拟内存与第 6 章 CPU 缓存的**最大区别**——CPU 缓存是访问时自动填入（硬件负责），虚拟内存的按需调页也是访问时调入，但由 OS 软件通过缺页故障驱动。

## 四、虚拟内存作为内存管理的工具（§9.4）

### 4.1 每进程独立的虚拟地址空间

每个进程有自己的页表，内核在上下文切换时切换 CR3 寄存器指向新进程的页表基址。因此：

- 两个进程可以同时使用相同的虚拟地址（如 `0x4004d0`），但映射到完全不同的物理页
- 进程 A 无法访问进程 B 的内存（除非显式共享）

**这是虚拟内存最重要的设计目标：让每个进程以为自己独占全部内存。**

### 4.2 典型进程内存布局

```
高地址
+----------------------------------+
|          内核空间（用户不可读）      |  ← 所有进程共享，仅内核态可访问
+----------------------------------+
|              栈 (stack)           |  ← 运行时向下增长 ↑；存放局部变量、返回地址
+----------------------------------+     匿名页，后备存储 = swap
|              共享库映射区域         |  ← libc.so 等加载在此
+----------------------------------+
|              堆 (heap)            |  ← 运行时向上增长 ↑；malloc/free 管理
+----------------------------------+     匿名页，后备存储 = swap
|      .bss 段（未初始化静态数据）     |  ← 程序加载时清零；初始映射到全局零页
+----------------------------------+
|      .data 段（已初始化静态数据）    |  ← 固定初始值来自 ELF 文件
+----------------------------------+
|      .rodata 段（只读数据）         |  ← 字符串常量、只读全局变量
+----------------------------------+
|      .text 段（代码/指令）          |  ← 只读、可执行
+----------------------------------+
|              保留/低地址区域         |
低地址
```

#### 各段的作用与生命周期

| 段名 | 存储内容 | 生命周期 | 可执行文件中占磁盘空间？ |
|:---:|---------|:------:|:---------------------:|
| `.text` | 机器指令（函数代码）| 整个运行期间 | **是** |
| `.rodata` | 字符串常量、`const` 全局变量 | 整个运行期间 | **是** |
| `.data` | 已初始化且初值非零的全局/静态变量 | 整个运行期间 | **是**（初始值存储在文件中）|
| `.bss` | 未初始化或零初始化的全局/静态变量 | 整个运行期间 | **否**（只记录大小，加载时映射到零页）|
| 堆 | 动态分配（`malloc` 等）| 分配→显式释放 | 运行时产生，不在文件中 |
| 栈 | 局部变量、调用帧、返回地址 | 函数调用期间 | 运行时产生，不在文件中 |

### 4.3 后备存储：Swap Space 与持久化文件

#### Swap Space

磁盘上的一块专用区域（分区或文件），作为**匿名页的后备存储**。

- **核心作用**：当物理内存不足时，将不常访问的匿名页换出到 swap，释放 DRAM
- **支持休眠**：将整个内存镜像写入 swap，实现"休眠到磁盘"
- **关键特点**：临时性（重启后内容失效）；速度远慢于 DRAM

#### 持久化文件

与 swap 相对的概念，用于存放**文件后备页的原始数据**：

| 数据类别 | 对应段 | 存储在哪个持久化文件？ |
|:--------|:----:|:---------------------|
| 可执行程序的机器指令 | `.text` | 可执行文件本身（如 `prog`）|
| 只读数据（常量）| `.rodata` | 可执行文件 |
| 已初始化全局/静态变量 | `.data` 的初始内容 | 可执行文件 |
| 动态链接共享库的代码与只读数据 | 共享库 `.text`/`.rodata` | 共享库文件（如 `libc.so`）|
| 通过 `mmap` 映射的普通文件数据 | 映射区域 | 被映射的普通文件 |

**关键点**：持久化文件**不存储**以下内容——`.bss`（只记录大小）、堆/栈数据（运行时产生）、修改过的 `.data`/`.bss`（COW 私有脏页已脱离原始文件，转而依赖 swap）。

#### 各段与后备存储的对应关系

| 段 | 持久化文件？ | 使用 swap？ |
|:--|:----------:|:---------:|
| `.text` | **是** | 基本不需要：干净页直接丢弃，下次从文件重读 |
| `.rodata` | **是** | 不需要 |
| `.data`（已初始化）| **是**（初始值来自文件）| **可能需要**：修改后变成脏页→需 swap |
| `.bss` | **否** | **是**（全为匿名页）|
| 堆 | **否** | **是** |
| 栈 | **否** | **是** |

## 五、虚拟内存作为内存保护的工具（§9.5）

每个 PTE 包含权限位，MMU 在每次地址翻译时自动检查：

### 5.1 PTE 中的权限位

| 位 | 全称 | 含义 |
|:--:|------|------|
| **SUP** | Supervisor | 1 = 内核模式才能访问该页；0 = 用户模式可访问 |
| **READ** | Read | 该页是否可读 |
| **WRITE** | Write | 该页是否可写 |
| **EXEC** | Execute | 该页是否可执行（NX bit）|

### 5.2 段级的保护

| 段 | SUP | WRITE | 保护效果 |
|:--:|:---:|:-----:|---------|
| `.text` | 0 | 0 | 用户态可读可执行，不可写 → 防止代码注入 |
| `.rodata` | 0 | 0 | 只读 → 防止修改字符串常量 |
| `.data` / `.bss` | 0 | 1 | 可读写，不可执行 → 数据区不能执行代码 |
| 栈 | 0 | 1 | 可读写，不可执行（NX）→ 栈缓冲区溢出不能直接执行注入代码 |
| 内核空间 | **1** | — | 用户态完全不可访问 → 保护内核 |

**违反权限 → 段错误（SIGSEGV）**，这是第 8 章 §8.6 的经典信号示例。

## 六、地址翻译 —— 端到端流程（§9.6–§9.7）

### 6.1 地址结构：虚拟地址与物理地址

```
虚拟地址 (VA)：
┌──────────────────────┬──────────────────┐
│    VPN (虚拟页号)     │   VPO (页内偏移)   │
│    n = p 位          │    p 位           │
│  TLBT   |   TLBI     |                  | 
└──────────────────────┴──────────────────┘

物理地址 (PA)：
┌──────────────────────┬──────────────────┐
│    PPN (物理页号)     │   PPO (页内偏移)  │
│   m - p 位           │   p 位           │
│      CT（cache tag） |   CI   |   CO    |                
└──────────────────────┴──────────────────┘

注意：PPO = VPO（页内偏移不变，只翻译页号！）
```

### 6.2 页表（Page Table）

页表是 VPN → PPN 的映射表，存储在物理内存中。每个进程有自己独立的页表，CR3 寄存器指向当前进程的页表基址。

**每个页表条目（PTE）的结构**：

```
PTE：
┌─────┬───┬─────────┬─────┬──────────────────┐
│Valid│...│权限位(RWSU)│Dirty│  PPN (物理页号)   │
└─────┴───┴─────────┴─────┴──────────────────┘

Valid = 1 → 页在物理内存中，PPN 有效
Valid = 0 → 页不在物理内存中（或未分配）
```

### 6.3 多级页表

由于 64 位地址空间太大（$2^{64}$ 字节），如果用单级页表——每个 PTE 8 字节 × $2^{52}$ 条目 = 天文数字的物理内存。因此使用**多级页表**，按需只为使用的区域创建下级表。

x86-64 使用**四级页表**，每级 9 位索引（$2^9 = 512$ 条目）：

```
                四级页表
虚拟地址 = | PML4 | PDPT | PD | PT | VPO |
            9 bits  9 bits 9bits 9bits 12 bits

（47 位虚拟地址空间，但地址实际只用低 48 位，高 16 位是符号扩展）
```

| 级别 | 名称 | 缩写 | 位数 |
|:---:|------|:---:|:---:|
| 1 | Page Map Level 4 | PML4 | 9 bits |
| 2 | Page Directory Pointer Table | PDPT | 9 bits |
| 3 | Page Directory | PD | 9 bits |
| 4 | Page Table | PT | 9 bits |
| — | 页内偏移 VPO | Offset | 12 bits (= 4KB 页)|

**为什么用多级页表？**
- 单级页表：必须为整个地址空间分配页表（即使大部分未使用）
- 多级页表：只为实际使用的区域分配——每个中间级表没被用到就不分配对应子表
- 如果 VPN-1 = 全 0 表示子表不存在（Valid = 0），直接缺页

### 6.4 TLB —— 页表的缓存

每次地址翻译都要查页表（走 DRAM）太慢。TLB（Translation Lookaside Buffer）是 CPU 内的 SRAM 缓存，缓存最近翻译过的 PTE。

```
TLB = 小容量的 SRAM（全相联或高相联度）
      缓存 <VPN, PTE> 映射关系

TLB 命中：1 个周期完成翻译
TLB 不命中：需查页表（可能多次访问 DRAM）→ 更新 TLB
```

### 6.5 完整的地址翻译流程

```
CPU 发出虚拟地址 VA
    │
    ▼
┌──────────┐
│ 查 TLB    │ ← SRAM，~1 周期
└──┬───────┘
   │
   ├── TLB Hit → 直接拿到 PPN + VPO → 物理地址 PA → 访问 L1 缓存
   │
   └── TLB Miss
         │
         ▼
   查多级页表（在 DRAM 中，需多次访存）
         │
         ├── PTE 有效（Valid=1）→ 取得 PPN，更新 TLB，返回 PA
         │
         └── PTE 无效（Valid=0）→ 缺页故障！
              OS 缺页处理程序：
              ├── 文件后备页 → 从磁盘 ELF/文件 读入 4KB 到 DRAM
              ├── 匿名页     → 分配零页 / 从 swap 换入
              ├── 更新 PTE（设为 Valid，填入 PPN）
              └── 返回，CPU 重新执行故障指令（这次命中）

拿到 PA 后，访问 L1 缓存（第 6 章）:
   L1 Hit (~4 cycles) → 数据返回 CPU
   L1 Miss → L2 → L3 → DRAM
```

### 6.6 缺页故障（Page Fault）

缺页故障是第 8 章 §8.1 "故障"异常的经典实例：

| 步骤 | 第 8 章对应 | 发生了什么 |
|:---:|:---------:|-----------|
| 1 | 故障异常 | CPU 检测到 PTE Valid=0，触发缺页异常 |
| 2 | 硬件×OS 协作 | CPU 保存状态、切换内核栈、跳转到缺页处理程序 |
| 3 | OS 处理 | OS 加载页面（从文件或 swap 到 DRAM）、更新 PTE |
| 4 | 返回当前指令 | 故障返回行为：**重新执行触发缺页的那条指令**（这次 PTE 有效，成功）|

## 七、内存映射 mmap（§9.8）

### 7.1 `execve` 如何加载程序

当调用 `execve("./prog", ...)` 时，内核做的是（不创建新进程，而是替换当前进程映像）：

1. 释放旧页表
2. 解析 ELF 文件（第 7 章），读取 segment 表
3. 将 ELF 的 `.text`、`.rodata`、`.data` 等段映射到虚拟地址空间（**不实际读入 DRAM**——只是建立页表映射，标记为 Valid=0）
4. 为 `.bss` 分配虚拟页，映射到零页
5. 创建初始栈（填充 `argc`、`argv[]`、`envp[]`）
6. 设置 `%rsp` 指向新栈顶，跳转到 `_start`

**第一次执行 `.text` 中的指令时**：触发缺页故障 → 内核从 ELF 文件读入对应页面到 DRAM → 更新 PTE → 重试指令。这就是"懒加载"——不访问就不加载。

### 7.2 mmap 系统调用

```c
void *mmap(void *start, size_t length, int prot, int flags,
           int fd, off_t offset);
```

将文件或匿名内存映射到虚拟地址空间：

| flags | 含义 |
|:-----:|------|
| `MAP_PRIVATE` | 私有映射——修改时触发 COW，不写回文件 |
| `MAP_SHARED` | 共享映射——修改对所有映射同一文件的进程可见 |
| `MAP_ANON` | 匿名映射——无对应文件，后备存储为 swap |

## 八、COW 写时复制机制

### 8.1 COW 的基本思想

**写时复制（Copy-on-Write, COW）**：多个进程共享同一块物理内存，直到某个进程试图写入时，系统才为它复制一个私有副本。

**核心价值**：大大减少了不必要的内存复制，节省物理内存，加快进程创建速度。

COW 本身是**缺页异常处理**和**物理页管理**的行为，**不直接触发磁盘 I/O**——但 COW 的结果决定后续换出行为（脏页→需 swap）。

### 8.2 COW 与 fork()

CSAPP 中 COW 最典型的应用是 **`fork()` 创建子进程**（第 8 章）：

```c
pid_t pid = fork();             // 第8章§8.4
if (pid == 0) {                 // 子进程
    // 父子都看到这里，但数据各自独立
}
```

**逐步流程**：

1. **`fork()` 后**：子进程被创建，**不复制父进程的任何物理页**。子进程的页表克隆父进程的页表，所有可写页（`.data`、`.bss`、堆、栈）在父子双方的页表中都被标记为**只读**。

2. **任一进程首次写入**（如子进程执行 `x++`）：
   - CPU 检测到页只读 → 触发**保护故障**（第 8 章 §8.1 故障异常）
   - 内核发现该页属于 COW 区域：
     - 分配一个新的物理页
     - 将旧页内容复制到新页
     - 修改触发进程的页表：指向新页，设为可读写
   - 进程重新执行写入指令 → 成功

3. **另一个进程继续使用自己那份副本**，互不干扰。

4. 此后若物理内存紧张：这个 COW 产生的私有页是脏匿名页 → 换出需写入 **swap space**。

**这就是为什么"fork 很快"**——它并不真的复制所有内存，只是复制了页表，真正的物理拷贝按需发生在写入时。除了 `.text` 和 `.rodata` 之外，`.data`、`.bss`、堆和栈在 `fork()` 后均用到了 COW 机制。

### 8.3 COW 与共享库

当一个共享库（如 `libc.so`）被多个进程加载时：
- 共享库的 **`.text` 段**（代码）纯只读，永久共享，无需 COW
- 共享库的 **`.data` 和 `.bss`**（全局变量）每个进程需要独立副本，但通过 COW 延迟复制：初始时所有进程页表指向同一物理数据页（只读），写入时触发 COW 创建私有副本

这正是动态链接器利用 COW 实现"多个进程共享同一份动态库的代码，但各自拥有独立数据"的经典方法。

### 8.4 COW 与 mmap 私有映射

`mmap` 以 `MAP_PRIVATE` 映射文件时，进程可以读文件内容，修改只在内存中进行、不写回文件。这一修改同样通过 COW 实现——多个进程映射同一文件时共享物理页，写入时触发 COW。

## 九、动态内存分配（§9.9–§9.10）

### 9.1 malloc 族函数

```c
void *malloc(size_t size);      // 分配 size 字节（不初始化）
void free(void *ptr);           // 释放 ptr 指向的块
void *calloc(size_t nmemb, size_t size);  // 分配 + 清零
void *realloc(void *ptr, size_t size);    // 调整已分配块大小
void *sbrk(intptr_t incr);      // 内核级：扩展或收缩堆的 brk 指针
```

`malloc` 返回的内存**不初始化**（值是垃圾桶里的旧数据），`calloc` 清零。`free` 后指针成为**悬挂指针**，不应再使用。`realloc` 可能在原地扩展，也可能另找新位置+复制旧数据+释放旧块。

### 9.2 堆的组织——隐式空闲链表

堆是进程虚拟地址空间中 `.bss` 上方、向高地址增长的区域。`malloc` 通过维护**块（block）**的链表来跟踪哪些区域已被分配、哪些空闲。

#### 块的基本结构：头部 + 有效载荷 + 填充

每个块（无论是已分配还是空闲）都由以下部分组成：

```
已分配块：
┌────────────┬──────────────────────────┬────────────┐
│  Header    │       Payload             │  Padding    │
│(size | a)  │    (malloc 返回的指针     │ (对齐填充)  │
│  4/8 字节  │      指向这里！)           │   可选      │
└────────────┴──────────────────────────┴────────────┘

空闲块：
┌────────────┬────────────┬────────────────┬────────────┐
│  Header    │  Footer    │    未使用的空间  │  可能还有   │
│(size | a=0)│(size | a=0)│                │  更多填充   │
└────────────┴────────────┴────────────────┴────────────┘
              ↑
       空闲块有 Footer，
       而已分配块一般没有（节省空间）
```

**Header（头部）**：记录块的大小（含头部本身）和分配标志位。最低 1 bit = `a`（allocated），其余高位 = size。由于块大小总是双字对齐（8 或 16 字节倍数），低位有冗余可以用来存标志——这是"偷 bit"的经典技巧。

```
Header 的一个字（8 字节示例）：
┌────────────────────────────────────┬───┐
│         size (块的总字节数)          │ a │
│         高 63 位                    │LSB│
└────────────────────────────────────┴───┘
  a=1: 已分配    a=0: 空闲
```

#### 隐式空闲链表的遍历

"隐式"意味着链接信息（下一个块在哪）**藏在块的 size 中**，不需要额外的指针。给定当前块指针 `p`：

```
下一个块 = p + p->size         （按 size 跳到下一个块）
上一个块 = p - prev->size      （需要 Footer 才知道上一个块的大小）
```

```
  堆中连续三个块：
  ┌──────────┬──────────┬──────────┬──────────┐
  │块1(size=32│块2(size=16│块3(size=48│          │
  │    a=1)   │    a=0)   │    a=1)   │   ...    │
  └──────────┴──────────┴──────────┴──────────┘
        ↑          ↑          ↑
       p          p+32      p+32+16
```

### 9.3 边界标记与合并

#### 为什么需要 Footer（边界标记）？

释放一个块时，需要检查**物理上相邻**的前一个和后一个块是否也是空闲的——如果都是空闲的，就应该合并成一个更大的空闲块，否则会产生外部碎片。Footer 存放的是**与 Header 完全相同的 size 和 a 标志**，使得从当前块可以向前找前一块的信息。

```
空闲块有 Footer：
┌──────────┬─────────────┬──────────┐
│ Header   │   未使用空间  │ Footer   │
│(size|a=0)│             │(size|a=0)│
└──────────┴─────────────┴──────────┘
            从下一个块的 Header 往后退一字 = 当前块的 Footer（在空闲时才有意义）
```

#### 合并（Coalescing）的四种情况

释放块 `p` 时，检查其前驱（prev，通过 Footer）和后继（next，通过 p+size）：

| 前驱 | 后继 | 操作 |
|:---:|:---:|------|
| 已分配 | 已分配 | 仅标记当前块为 `a=0`，不合并 |
| 已分配 | 空闲 | `p` 与后继合并（新块 = p + next，size累加）|
| 空闲 | 已分配 | `p` 与前驱合并（prev + p）|
| 空闲 | 空闲 | **全部合并**（prev + p + next）→ 最大努力减少外部碎片 |

**合并为什么需要 Footer？** 要判断"前一个块是否空闲"，必须读前一个块的 size 和 a 标志——这些信息恰好在前一块的 Footer 中。从 `p` 的 Header 往前退一字就是前一块的 Footer（如果前一块是空闲的）：

```c
// 判断前驱是否空闲
prev_footer = (unsigned int *)((char *)p - sizeof(int));
prev_alloc  = prev_footer & 1;
prev_size   = prev_footer & ~1;
```

#### 立即合并 vs 延迟合并

| 策略 | 做法 | 优缺点 |
|:---:|------|--------|
| **立即合并** | 每次 `free` 立即检查并合并相邻空闲块 | 简单，但可能产生"乒乓效应"——刚合并的块马上又被切开分配给下一个 `malloc` |
| **延迟合并** | `free` 时不立即合并，等到 `malloc` 请求失败（找不到足够大的空闲块）时才扫描堆并合并 | 减少反复合并-切开的开销，但实现更复杂 |

### 9.4 分配器的实现：从隐式链表到显式链表

#### 9.4.1 隐式空闲链表 + 首次适配

最朴素的实现：在堆上顺序扫描每个块（用隐式链表遍历），找第一个足够大的空闲块。

```
malloc(size) 的隐式链表版伪代码：
  需要的块大小 = align(size + header_size)
  p = heap_start
  while (p != heap_end):
      if (p->a == 0 && p->size >= needed):
          切分：如果剩余部分 ≥ 最小块大小 → 切出一个空闲子块
          标记 p->a = 1
          返回 p + header_size (payload 指针)
      p = p + p->size
  如果没找到 → 用 sbrk 扩展堆，返回新块
```

**切分（Splitting）**：当一个空闲块大于请求大小时，将它切为两块——前一块分配给用户，后一块保持空闲。切分后剩余必须 ≥ 最小块大小（否则不切，整块给用户，减少外部碎片）。

**复杂度**：`malloc` $O(n)$（n=堆中块的数量），`free` $O(1)$（仅标记 a=0 + 可能合并邻居）。

#### 9.4.2 显式空闲链表

隐式链表的痛点：`malloc` 需要扫描**所有**块（包括已分配块）来找空闲块。显式空闲链表只在**空闲块之间**维护双向链表指针，跳过已分配块：

```
空闲块结构（含显式链表指针）：
┌──────────┬──────────────┬──────────────┬──────────┬──────────┐
│ Header   │   pred (前驱) │  succ (后继)  │   ...    │ Footer   │
│(size|a=0)│  指向前一个   │  指向后一个   │ 剩余空间 │(size|a=0)│
└──────────┴──────────────┴──────────────┴──────────┴──────────┘
```

**`malloc` 的变化**：只遍历空闲链表（跳过所有已分配块），将首次适配复杂度从 $O(\text{总块数})$ 降低到 $O(\text{空闲块数})$。

**`free` 的变化**：释放时不仅处理合并，还要将新空闲块插入到显式空闲链表中。插入策略影响性能：

| 插入策略 | 做法 | 效果 |
|:------:|------|------|
| **LIFO（后进先出）** | 新释放的块插到链表头部 | 快（O(1)插入），但释放模式会导致碎片集中在头部 |
| **地址序** | 按堆地址排序插入 | 合并效率高（相邻空闲块在链表中也相邻），但插入需 O(n) 搜索 |

#### 9.4.3 分离空闲链表（Segregated Free Lists）

显式链表的进一步优化：将空闲块按大小分成多个**类（size class）**，每类维护一条独立的显式空闲链表。

```
大小类划分（类似伙伴系统）：
  (0,   16]   → 链表 1   （微小块）
  (16,  32]   → 链表 2
  (32,  64]   → 链表 3
  (64, 128]   → 链表 4
  (128, 256]  → 链表 5
  (256, 512]  → 链表 6
  (512, ∞)    → 链表 7   （大块，直接 mmap）
```

**`malloc` 流程**：
1. 根据请求大小定位对应的 size class
2. 在该类的链表中找空闲块（首次适配或最佳适配）
3. 若该类无空闲块 → 去下一个更大的类中找
4. 若所有类都没有 → `sbrk` 扩展堆，或将新分配的块放入对应 size class

**优势**：
- 小请求从小块池分配（快速，碎片少），大请求从大块池分配（不用频繁切分）
- 每个 size class 内的空闲块大小相近 → 搜索效率接近 $O(1)$
- 这就是 glibc 的 `ptmalloc` (pthreads malloc) 实际采用的设计

### 9.5 碎片问题——深入理解

| 类型 | 成因 | 说明 | 举例 |
|:---:|------|------|------|
| **内部碎片** | 对齐约束 + 最小块限制 + Header/Footer 开销 | 分配器给出的块 > 用户请求的大小 | `malloc(3)` → 实际占用 16 字节（对齐到 8 + Header 8B = 16），浪费 5 字节 |
| **外部碎片** | 多次 `malloc`/`free` 后，空闲总空间够但没有**一个连续块**满足大请求 | 需要合并来缓解 | 释放两个 32B 的相邻块 → 合并成 64B → 才能满足 64B 的请求 |

**外部碎片是分配器设计的主要挑战**——不可能完全消除，只能通过聪明的合并策略和分离链表来缓解。

### 9.6 分配器设计的完整迭代总结

| 迭代 | 设计 | malloc 复杂度 | free 复杂度 | 核心改进 |
|:---:|------|:----------:|:--------:|---------|
| v1 | 隐式空闲链表 + 首次适配 | $O(\text{总块数})$ | $O(1)$ | 最简洁的实现 |
| v2 | v1 + 边界标记 + Footer | 同上 | $O(1)$（加合并）| 释放时合并相邻块，减少外部碎片 |
| v3 | v2 + 显式空闲链表 | $O(\text{空闲块数})$ | $O(1)$（加链表操作）| 跳过已分配块，搜索更快 |
| v4 | v3 + 分离链表 | 接近 $O(1)$ | $O(1)$ | 按大小分类，小请求从小池取，碎片更少 |
| 工业级 | 多线程优化 + mmap大块 + 延迟合并 | $O(1)$ 摊销 | $O(1)$ | glibc ptmalloc 的实际方案 |

### 9.7 垃圾收集简述（§9.10）

> CSAPP §9.10 仅作介绍性内容，考试通常不重点考察。

**基本思想**：自动回收不再被引用的堆内存（程序员无需手动 `free`）。通过维护一个**可达图（reachability graph）**——从根节点（寄存器、栈、全局变量）出发，能到达的所有已分配块都是"活着的"，其余是"垃圾"，可回收。

**标记-清除算法**：
1. **标记阶段**：从根节点 DFS/BFS 遍历所有可达块，标记为"活"
2. **清除阶段**：扫描堆，释放所有未标记的块

> C 语言本身不支持垃圾收集（程序员必须手动 `free`），但 Boehm GC 等第三方库可以给 C/C++ 提供 GC 支持。

## 十、重点总结 & 常见考题类型

### 10.1 核心原理一句话

> **虚拟内存通过"每个进程独立页表 + MMU 地址翻译 + 缺页驱动加载"三个机制，给每个进程一个独立、连续、完整的虚拟地址空间。物理内存是磁盘的缓存，缺页是透明翻译失败后的补救。**

### 10.2 必会技能清单

1. **虚拟地址 → 物理地址翻译**：给定 VPN/VPO 和页表，计算 PA
2. **多级页表地址分解**：将 VA 按 PML4/PDPT/PD/PT/Offset 拆解
3. **PTE 字段解读**：Valid/Dirty/权限位/PPN 各代表什么
4. **缺页流程**：从 TLB miss → 查页表 → PTE 无效 → 故障 → OS 处理 → 重试
5. **进程内存布局**：各段的顺序（从低地址 .text → 高地址 栈）、增长方向、后备存储
6. **文件后备页 vs 匿名页**：各段的分类、换出行为区别
7. **COW 机制**：fork 后父子如何共享页、写入时如何触发缺页分离
8. **mmap 与 execve**：程序加载的懒加载过程

### 10.3 与第 6 章 + 第 8 章的三角关系

```
第6章 缓存
  │  缓存块→页面；L1/L2/L3→DRAM；主存→磁盘
  │  命中/不命中→页命中/缺页；写回→脏页写回
  │
  ▼
第9章 虚拟内存 ──────── 第8章 ECF
   缺页 = 故障异常              │
   mmap = 懒加载           COW = 保护故障
   页表权限 = SIGSEGV       fork = COW 用例
   TLB = 缓存              上下文切换 = 切换 CR3
   CR3 = 页表基址           SIGSEGV = 非法内存访问
```
